# 디자인 퀄리티 표준화 — 분석 보고서

## 1. 현재 문제

프로토타입 수준의 디자인밖에 안 나오는 이유:

```
현재: "홈 화면 디자인 해주세요" → AI가 generic한 UI 생성
이상: "이 토큰, 이 컴포넌트 규격, 이 원칙으로" → 프로덕션 퀄리티
```

**핵심 원인: AI에게 "무엇을 만들어라"만 전달하고, "어떤 기준으로 만들어라"를 전달하지 않음.**

## 2. Supanova Design Skill 분석

GitHub: `uxjoseph/supanova-design-skill`

### 구조

```
supanova-design-skill/
├── taste-skill/     ← 핵심 디자인 엔진 (레이아웃, 타이포, 컬러, 모션)
├── soft-skill/      ← 프리미엄 미학 ($150k 에이전시 퀄리티)
├── redesign-skill/  ← 기존 페이지 업그레이드
└── output-skill/    ← 완성도 강제 (플레이스홀더 차단)
```

### 퀄리티를 높이는 핵심 기법

**① 디자인 변수 4개로 톤 제어**

| 변수 | 범위 | 역할 |
|------|------|------|
| DESIGN_VARIANCE | 1~10 | 레이아웃 비대칭/창의성 수준 |
| MOTION_INTENSITY | 1~10 | 애니메이션 복잡도 |
| VISUAL_DENSITY | 1~10 | 정보 밀도 vs 여백 비율 |
| LANDING_PURPOSE | enum | 전환/브랜드/포트폴리오/SaaS |

→ 프로젝트 시작 시 이 4개 값을 설정하면 AI의 디자인 방향이 잡힘.

**② Anti-Pattern 명시적 금지**

- 금지 폰트: Inter, Roboto, Arial (generic함)
- 금지 아이콘: 두꺼운 스트로크 아이콘
- 금지 컬러: 보라색 "AI" 그라데이션, 네온, 순수 검정(#000000)
- 금지 표현: "혁신적인" 같은 번역투 클리셰

→ "하지 마라"를 명확히 하면 generic함이 줄어듦.

**③ 프리미엄 컴포넌트 패턴**

- Double-Bezel 카드: 외부 껍질(알루미늄) + 내부 코어(글래스) 분리
- Floating Glass Navigation: `backdrop-blur-xl` + `bg-white/10`
- 모션: 단일 easing `cubic-bezier(0.16, 1, 0.3, 1)` 통일

→ "어떻게 만들어라"가 구체적일수록 퀄리티가 올라감.

**④ 출력 검증 체크리스트**

만든 후 반드시 확인: 폰트 로드, 아이콘 일관성, 금지 패턴 사용 여부, 모바일 터치 영역, 레이아웃 다양성.

## 3. 우리 파일('디자인 관련 내용 정리하기.md')과 비교

| 항목 | 우리 파일 | Supanova |
|------|----------|----------|
| **디자인 토큰** | 색상 16개, 타이포 8단계, 간격 11단계 — 매우 상세 | 변수 4개로 방향 제어 — 추상적 |
| **컴포넌트 규격** | Button, Input, Card, Modal 등 구체적 px값 | 프리미엄 패턴(Double-Bezel 등) 중심 |
| **판단 원칙** | 8가지 원칙 (CTA 1개, 계층 3단계 등) | Anti-Pattern 금지 목록 |
| **출력 형식** | ASCII 다이어그램 + 상태별 UI + 체크리스트 | 단일 HTML 파일 |
| **적용 대상** | 모바일 앱 + 웹 + 관리자 웹 | 랜딩 페이지 특화 |

**우리 파일이 더 체계적이고 범용적입니다.** Supanova는 랜딩 페이지에 특화되어 있어 앱 디자인에는 직접 적용이 어렵지만, "Anti-Pattern 금지"와 "디자인 변수로 톤 제어"는 배울 점이 있습니다.

## 4. 디자인 퀄리티를 높이기 위해 필요한 것

### 프로토타입 → 프로덕션 퀄리티 차이

```
프로토타입:  "버튼이 있다" → 아무 버튼
프로덕션:   "이 크기, 이 색상, 이 radius, 이 hover 상태의 버튼" → 구현 가능한 버튼
```

### 필요한 정보 3계층

```
Layer 1. 디자인 토큰 (프로젝트 독립적 — 회사 표준)
  → 색상 체계, 타이포 스케일, 간격, 반경, 그림자, z-index
  → 어떤 프로젝트든 이 구조를 따름

Layer 2. 프로젝트 디자인 시스템 (프로젝트별)
  → Layer 1의 구조에 프로젝트 고유 값을 채움
  → 예: PACE는 Spotify 다크 테마, on-your-lunch는 배민 화이트 테마

Layer 3. 화면별 상세 명세 (화면별)
  → Layer 2 위에 화면 구조, 컴포넌트 배치, 인터랙션 정의
```

### 현재 빠진 것

| 계층 | 현재 상태 | 필요한 것 |
|------|----------|----------|
| Layer 1 | `디자인 관련 내용 정리하기.md`에 있음 (미적용) | `rules/design/`에 표준으로 반영 |
| Layer 2 | 프로젝트마다 즉흥적으로 결정 | 프로젝트 시작 시 디자인 시스템 문서 필수 작성 |
| Layer 3 | screen 문서에 구조만 있음 | 컴포넌트 규격 + 상태 + 토큰 참조까지 포함 |

## 5. 표준화 제안

### 제안 1: 디자인 토큰 표준을 rules에 반영

`디자인 관련 내용 정리하기.md`의 내용을 `rules/design/`에 **회사 표준**으로 등록. 모든 프로젝트에서 이 구조를 따르되, 값은 프로젝트별로 다르게.

```
rules/design/
├── design-rules.md          ← 기존 (도구, 4상태, 역할 분리)
├── design-tokens.md         ← 신규: 토큰 구조 표준 (색상/타이포/간격/반경/그림자)
├── design-principles.md     ← 신규: 판단 원칙 8가지
└── component-spec.md        ← 신규: 컴포넌트 명세 작성 규칙
```

### 제안 2: 프로젝트 시작 시 디자인 시스템 문서 필수

`/init` 또는 Stage 3(디자인) 진입 시, 디자인 시스템 문서를 먼저 작성:

```
.docs/005_design/design-system.md
  ├── 이 프로젝트의 색상 값 (토큰 구조에 값 채움)
  ├── 이 프로젝트의 폰트
  ├── 이 프로젝트의 컴포넌트 기본 규격
  └── 이 프로젝트의 UX 벤치마크 패턴
```

### 제안 3: Anti-Pattern 목록 추가 (Supanova에서 차용)

프로젝트별로 "이건 하지 마라" 목록:
- 금지 폰트, 금지 컬러, 금지 패턴
- 프로젝트 컨셉과 맞지 않는 것을 명시적으로 차단

### 제안 4: 디자인 산출물 체크리스트 강화

화면 완성 시 검증:
- 토큰만 사용했는가? (임의 값 없음)
- 4가지 상태가 모두 있는가?
- 터치 영역 44px 이상인가?
- 벤치마크와 일관되는가?
- Anti-Pattern에 해당하는 것이 없는가?

## 6. 요약

```
퀄리티가 안 나오는 이유:
  "디자인 해주세요" (방향 없음) → generic 결과

퀄리티를 올리는 방법:
  Layer 1 (토큰 표준) + Layer 2 (프로젝트 시스템) + Layer 3 (화면 명세)
  + Anti-Pattern 금지 + 체크리스트 검증

표준화할 것:
  1. 디자인 토큰 구조 → rules/design/에 회사 표준
  2. 프로젝트 디자인 시스템 → 필수 산출물로 등록
  3. Anti-Pattern 목록 → 프로젝트별 설정
  4. 산출물 체크리스트 → 강화
```

PO님, 이 방향으로 진행할지 결정해주시면 반영하겠습니다!
