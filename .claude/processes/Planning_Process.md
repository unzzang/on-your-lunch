# 기획 프로세스 (Stage 2)

> 적용 범위: 서비스 정의부터 기술 설계까지
> 책임자: PO (의사결정), 위더 (기획 문서 작성)
> 선행 조건: Stage 1 Go 결정
> 스킬: `/spec` (기능 명세서), `/decision` (DR 기록), `/minutes` (회의록)

---

## 1. 역할

서비스를 정의하고, 기능을 설계하고, 기술 아키텍처를 확정하여 개발 착수 가능한 상태를 만든다.

---

## 2. 프로세스

```
Gate 1. 서비스 정의
    ↓ PO 승인
Gate 2. 기능 설계
    ↓ PO 승인
Gate 3. 기술 설계
    ↓ PO 승인 → Stage 3 진입
```

---

### Gate 1. 서비스 정의

| 활동 | 담당 | 산출물 위치 |
|------|------|-----------|
| 타겟 사용자 정의 | 위더 + PO | `docs/004_planning/specs/` |
| 핵심 가치 + 차별점 | 위더 + PO | `docs/004_planning/specs/` |
| BM 설계 | 위더 + PO | `docs/004_planning/specs/` |
| 검증 | 송현아 | 피드백 |

**완료 기준:**

- [ ] 서비스 정의 문서 작성 완료
- [ ] 검증 통과 (송현아)
- [ ] PO 승인 → DR 기록 (`docs/001_decision-making/`)

---

### Gate 2. 기능 설계

| 활동 | 담당 | 산출물 위치 |
|------|------|-----------|
| 기능 맵 작성 | 위더 | `docs/004_planning/feature-map.md` |
| 기능 명세서 (기능별) | 위더 (`/spec`) | `docs/004_planning/specs/` |
| 화면 설계 문서 | 위더 | `docs/004_planning/screen/` |
| QA 관점 리뷰 | 진도준 | 피드백 |
| 검증 | 송현아 | 피드백 |

**완료 기준:**

- [ ] 기능 맵 작성 완료 (MVP 범위 확정)
- [ ] 기능별 명세서 전체 작성 완료
- [ ] 화면별 설계 문서 전체 작성 완료
- [ ] QA 리뷰 반영
- [ ] 검증 통과 (송현아)
- [ ] PO 승인 → DR 기록

---

### Gate 3. 기술 설계

| 활동 | 담당 | 산출물 위치 |
|------|------|-----------|
| 아키텍처 설계 | 이명환 | `docs/004_planning/tech-spec/` |
| ERD 설계 | 이명환 | `docs/004_planning/erd/` |
| API 설계 | 이명환 | `docs/004_planning/tech-spec/` |
| 데이터 설계 | 톰크루즈 | `docs/004_planning/tech-spec/` |
| 검증 | 송현아 | 피드백 |

아키텍처 설계와 데이터 설계는 병렬 수행 가능. `.claude/references/architecture-patterns.md` 참조.

**완료 기준:**

- [ ] 아키텍처 설계 완료 (스택, 패턴 확정)
- [ ] ERD 완료 (테이블, 관계, 인덱스)
- [ ] API 설계 완료 (엔드포인트, 요청/응답)
- [ ] 데이터 설계 완료 (수집 항목, KPI, 이벤트)
- [ ] 검증 통과 (송현아)
- [ ] PO 승인 → DR 기록
