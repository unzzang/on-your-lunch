# AI 협업 프로젝트 업무 플로우 가이드

> 최종 수정: 2026-03-28
> PACE 프로젝트에서 실전 검증한 워크플로우. 다른 프로젝트에도 재사용 가능.
>
> **전체 서비스 개발 프로세스(Stage 0~6)의 상세는 `Service_Development_Process.md` 참조.**
> 이 문서는 워크플로우의 실행 가이드이며, 프로세스 문서는 각 Stage의 절차·완료 기준·산출물을 정의한다.

---

## Phase 1. 프로젝트 셋업

### 1-1. 팀 구성
- PO(사람) + AI 에이전트 팀 구성
- 역할별 에이전트 정의: 기획, 디자인, 프론트엔드, 백엔드, QA, 데이터 등
- 각 에이전트의 AGENT.md 작성 (정체성, 전문 분야, 담당 범위, 협업 접점)
- **산출물:** `.claude/agents/{이름}/AGENT.md`

### 1-2. 커뮤니케이션 구조 확정
- 허브형 (메인 기획자가 중재) vs 직접 소통 vs 하이브리드
- 문서 핸드오프 규칙 (품질 확인 주체, 전달 방식)
- **산출물:** 의사결정 기록 (DR)

### 1-3. 도구 환경 구축
- 디자인 도구 (Pencil MCP 등) 설정 및 테스트
- Skill 생성 (프로세스 스킬 + 에이전트 호출 스킬)
- 디자인 시스템 기초 셋업 (컬러, 타이포, 기본 컴포넌트)
- **산출물:** `.claude/skills/`, `design/design-guide.pen`

### 1-4. 문서 체계 수립
- 회의록 작성 규칙 (위치, 포맷, 번호 체계)
- 의사결정 기록 체계 (DR-XXX)
- 폴더 구조 확정
- **산출물:** 의사결정 기록 (DR)

## Phase 2. 서비스 기획

### 2-1. 서비스 정의
- 누구를 위한 서비스인가 (타겟 사용자)
- 어떤 문제를 해결하는가 (핵심 페인포인트)
- 핵심 가치는 무엇인가 (차별점, 경쟁 우위)
- 비즈니스 모델 (수익 구조)
- 팀원 이해도 확인 — 각 에이전트에게 질문하여 방향성 일치 검증
- **산출물:** 기획 문서, 메모리 저장

### 2-2. 기능 목록 (Feature Map)
- 전체 기능을 카테고리별로 나열 (Auth, Home, Search 등)
- 각 기능별 브레인스토밍 — PO와 1:1 질문/답변 방식으로 구체화
- 핵심 기능부터 시작, 점차 확장
- **QA 리뷰 필수:** 빠진 기능, 모호한 정의, 엣지 케이스, 플로우 끊김, MVP 범위 점검
- MVP 범위 확정 (O / △ / 이후)
- **산출물:** `planning/feature-map.md`

### 2-3. 화면 설계 (와이어프레임)
- Feature Map 기반으로 핵심 화면부터 디자인
- 순서: 핵심 플로우 → 서브 화면 → 예외 화면
- PO 확인/피드백 → 반영 → 재확인 사이클
- **산출물:** `design/pace-{플랫폼}-{화면명}.pen`, `design/images/`

### 2-4. 기능 명세서 (상세 스펙)
- 화면 설계와 동시 진행 — 화면을 보면서 "이 버튼 누르면 뭐가 되지?" 구체화
- 각 기능의 입력/출력, 조건 분기, 에러 처리, 경계값 정의
- **산출물:** `planning/specs/` (기능별 문서)

### 2-5. 플랫폼별 Feature Map
- 모바일 앱 (핵심)
- PC 웹 (모바일과 동일 or 차이점 정의)
- 관리자 웹 (운영 도구, 대시보드)
- **산출물:** 플랫폼별 feature-map

## Phase 3. 기술 설계

### 3-1. 데이터 스키마 & ERD
- 기능 명세 기반 테이블 설계
- 관계도 (ERD) 작성
- 백엔드 아키텍트(명환) 주도
- **산출물:** `planning/data-schema.md`

### 3-2. API 설계
- 엔드포인트 목록
- 요청/응답 형식
- 인증/권한 정책
- **산출물:** `planning/api-spec.md`

### 3-3. 기술 스택 선정

- 클라우드 (AWS / GCP 등)
- **산출물:** 의사결정 기록 (DR)

### 3-4. 시스템 아키텍처
- 전체 시스템 구성도
- 서버 구조, 배포 방식
- **산출물:** `planning/architecture.md`

## Phase 4. 개발 & QA

### 4-1. 개발 환경 셋업
- 프로젝트 초기화, 폴더 구조
- CI/CD 파이프라인
- 개발/스테이징/프로덕션 환경

### 4-2. 기능 개발
- MVP 기능 우선순위 기반 스프린트
- 프론트엔드 ↔ 백엔드 병행
- 디자인 스펙 → 개발 핸드오프 (위더 경유)

### 4-3. QA & 테스트

**표준 QA 프로세스를 따른다.** 상세: `.docs/planning/qa/QA_Process.md`

- Phase 1: 정적 검증 (빌드, 테스트, 린트, 타입 체크)
- Phase 2: 통합 검증 (API 계약, 환경 정합성, 공유 타입)
- Phase 3: 기능 검증 (스모크 테스트, 핵심 플로우, 에러 시나리오)
- Phase 4: 수정 재검증 (버그 재현 확인, 리그레션, QA 최종 승인)

QA는 서비스가 사용자에게 제공되기 전 최종 관문이다. QA 통과 판정 없이 릴리스하지 않는다.

## Phase 5. 론칭 & 운영

### 5-1. 론칭 준비
- 앱스토어 심사 준비 (탈퇴 기능, 약관, 개인정보 처리방침 등)
- 운영팀 콘텐츠 준비 (초기 플레이리스트)
- 관리자 웹 운영 준비

### 5-2. 론칭 후 운영
- 사용자 데이터 모니터링
- 피드백 수집 → 우선순위 정리
- 이후 버전 기능 개발

## 핵심 원칙

### 회의록은 즉시 작성
- 의사결정, 정책 변경, 액션 아이템 발생 시 반드시 기록
- 다음 세션에서 이전 맥락을 잃지 않기 위한 핵심 장치

### QA 리뷰는 기획 단계에서부터
- 기능 목록 완성 후 QA 리뷰 → 빠진 기능, 엣지 케이스 조기 발견
- 개발 후 QA보다 기획 중 QA가 수정 비용이 훨씬 저렴

### MVP는 과감하게 줄인다
- "있으면 좋은 것" vs "없으면 안 되는 것" 구분
- Zero-Setup 철학과 충돌하는 기능은 이후로

### 화면을 보면서 기획한다
- 문서만으로 기획하면 추상적. 화면을 그려야 구체적 질문이 나옴
- Feature Map → 화면 설계 → 기능 명세 보완 순서가 효율적

### 팀원 이해도를 확인한다
- 에이전트에게 "이 서비스가 뭐하는 건지 설명해보세요" 질문
- 방향성이 어긋나면 조기 교정

## Skill/Agent 사용 가이드

> 추가일: 2026-03-20
> 이 섹션은 PACE 프로젝트의 AI 에이전트 팀 운영 방법을 설명합니다.

### 기본 구조

```
PO (사용자)
  ↕
위더 (Claude Code 본체 — 시니어 기획자 + 중재자)
  ↕
에이전트들 (jisu, jack, tom-hardy, myunghwan, insoo, dojun, tom-cruise, ...)
```

모든 소통은 위더를 중심으로 한 **허브-스포크 구조**로 운영됩니다.

### Skill의 두 가지 유형

#### 1. 에이전트 호출 Skill

특정 역할의 에이전트를 소환하여 작업을 수행합니다.

**사용법:**
```
/design 온보딩 화면 디자인해주세요
/frontend 홈 화면 구현해주세요
/backend 사용자 엔티티 ERD 설계해주세요
```

**작동 원리:**
1. PO가 `/design` 입력
2. `.claude/skills/design/SKILL.md`가 로드됨
3. `agent: jisu` 설정에 따라 jisu 에이전트가 소환됨
4. jisu는 자신의 AGENT.md에 정의된 역할·성격으로 작업 수행
5. 프로젝트 컨텍스트는 CLAUDE.md에서 자동으로 로드됨

#### 2. 프로세스 Skill

문서 산출물의 형식과 규칙을 정의합니다. 위더가 직접 실행하거나, 에이전트가 참조합니다.

**위더가 직접 실행:**
```
/minutes                    → 현재 대화의 회의록 작성
/decision                   → 확정된 의사결정 기록
/spec 온보딩               → 온보딩 기능 명세서 초안 생성
/review 온보딩             → 온보딩 관련 문서 리뷰
/research 학습 플래너 시장  → 시장 심층 리서치
```

**에이전트가 참조:**
에이전트가 작업 중 명세서나 의사결정 기록이 필요할 때, 해당 프로세스 Skill 파일을 읽고 동일한 템플릿으로 작성합니다. 이를 통해 누가 만들든 산출물 형식이 일관됩니다.

예시 흐름:
```
PO: /backend 알림 시스템 설계해주세요
  → myunghwan 에이전트 소환
  → 설계 중 의사결정 필요 발견
  → .claude/skills/decision/SKILL.md 읽기 → 동일한 DR 템플릿으로 기록
  → 산출물 일관성 확보
```

### 에이전트 관리

#### 새 에이전트 추가
```
/add-agent sophia 서비스전략가 "초기 BM·서비스 아이디에이션"
```

자동으로:
1. `.claude/agents/sophia/AGENT.md` 생성
2. `.claude/skills/strategy/SKILL.md` 생성
3. CLAUDE.md 팀 테이블에 행 추가

#### 에이전트 제거
```
/remove-agent sophia
```

자동으로:
1. Agent/Skill 파일 삭제
2. CLAUDE.md 팀 테이블에서 행 제거
3. 다른 파일은 역할 기반 참조이므로 수정 불필요

### 다른 프로젝트에서 사용하기

이 Skill/Agent 구조는 프로젝트에 독립적으로 설계되었습니다.

#### 새 프로젝트에 적용하는 방법

1. `.claude/` 폴더 전체를 새 프로젝트로 복사
2. 새 프로젝트에 맞는 `CLAUDE.md` 작성 (프로젝트 개요, 도메인 용어, 기술 스택, 문서 구조)
3. 필요 없는 에이전트는 `/remove-agent`로 제거, 필요한 에이전트는 `/add-agent`로 추가
4. 완료 — Agent/Skill 파일 내부는 수정할 필요 없음

#### 프로젝트별로 달라지는 것 vs 고정되는 것

| 항목 | 프로젝트별 변경 | 고정 (재사용) |
|---|---|---|
| CLAUDE.md | O 매번 새로 작성 | |
| Agent 파일 (AGENT.md) | | O 역할·성격 그대로 사용 |
| 에이전트 호출 Skill | | O 그대로 사용 |
| 프로세스 Skill (템플릿) | | O 그대로 사용 |
| 팀 구성 | O CLAUDE.md 테이블로 관리 | |
| 참조 문서 경로 | O CLAUDE.md에 명시 | |
