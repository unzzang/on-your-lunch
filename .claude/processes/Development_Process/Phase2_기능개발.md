# Phase 2. 기능 단위 개발

> 기능 수만큼 반복. 기능 명세서 1개 = 개발 단위 1개.

## 2-0. 기능 검증 체크리스트 작성 (Phase 2 시작 전 필수)

**기능 개발을 시작하기 전에** 기능 검증 체크리스트를 먼저 만든다. 이 체크리스트가 없으면 Phase 2를 시작하지 않는다.

**작성 기반 (프로젝트 기획 산출물):**

- `docs/004_planning/specs/` — 기능 명세서 (기능별 동작 규칙, 에러 케이스)
- `docs/004_planning/screen/` — 화면 설계 (화면별 인터랙션, 상태 분기)
- `docs/004_planning/feature-map.md` — 기능 맵 (전체 기능 목록)

**절차:**

1. 기능 명세서의 각 기능을 읽는다
2. 화면 설계의 각 인터랙션을 읽는다
3. 모든 기능을 "사용자 액션 → 기대 결과" 형태로 변환한다
4. `docs/004_planning/{프로젝트명}_기능검증_체크리스트.md`에 작성한다

**형식:**

```markdown
# 기능 검증 체크리스트

## 화면 1: {화면명}
- [ ] {기능}: {사용자 액션} → {기대 결과}

## 화면 2: {화면명}
- [ ] ...
```

**완료 기준:**

- [ ] 기능 명세서의 모든 기능이 체크리스트에 포함됨
- [ ] 화면 설계의 모든 인터랙션이 체크리스트에 포함됨
- [ ] 각 항목이 "사용자 액션 → 기대 결과" 형태로 작성됨

## 기능 개발 사이클

각 기능마다 아래 사이클을 반복한다. 개발 중 체크리스트의 해당 항목을 체크하며 진행한다.

```
기능 N:
  2-1. 백엔드 API 구현 + 검증
  2-2. 프론트 UI + 인터랙션 구현 + 브라우저 검증
  2-3. 프론트↔백엔드 연동 확인
  → 체크리스트에서 기능 N 항목 전체 체크
  → 기능 N+1로 이동
```

## 2-1. 백엔드 기능 개발

**절차:**

1. shared-types의 요청/응답 타입을 기반으로 controller + service + DTO를 구현한다
2. 구현 후 curl 또는 Swagger로 정상 응답을 확인한다
3. 에러 케이스를 테스트한다 (잘못된 입력, 미인증, 중복, 존재하지 않는 리소스)
4. 스키마 필드가 빠짐없이 사용되는지 확인한다

**검증 방법:**

```bash
# 정상 케이스
curl -s -X POST http://localhost:3000/v1/엔드포인트 \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"필드": "값"}' | python3 -m json.tool

# 에러 케이스
curl -s -X POST http://localhost:3000/v1/엔드포인트 \
  -d '{"잘못된필드": ""}' | python3 -m json.tool
```

**완료 기준:**

- [ ] 정상 요청 → `{ success: true, data: {...} }` 반환
- [ ] 에러 요청 → `{ success: false, error: { code, message } }` 반환
- [ ] shared-types 응답 구조와 실제 응답이 일치

## 2-2. 프론트엔드 기능 개발

**절대 규칙: 브라우저에서 확인하지 않은 기능은 미완료.**

**절차:**

1. dev 서버를 실행한다 (`pnpm web:dev`)
2. UI를 구현한다 (컴포넌트, 페이지, 스타일)
3. **브라우저에서 렌더링을 확인한다** — 화면이 정상적으로 보이는가
4. 인터랙션을 구현한다 (클릭 핸들러, 상태 변경, API 호출 연결)
5. **브라우저에서 인터랙션을 확인한다** — 클릭하면 동작하는가
6. **결과 반영을 확인한다** — API 응답이 화면에 반영되는가
7. 에러/로딩/빈 상태를 구현하고 확인한다
8. 기능 완료 → 다음 기능으로 이동

**기능 유형별 브라우저 확인 항목:**

| 기능 유형 | 확인할 인터랙션 |
|----------|--------------|
| 필터 | 클릭 → 데이터 변경 → 화면 갱신. 다른 필터 클릭 → 이전 해제. 초기화. |
| 버튼 액션 | 클릭 → API 호출 → 성공 시 UI 변경. 실패 시 에러. 로딩 중 비활성화. |
| 폼 입력 | 입력 → 유효성 검증 → 제출 → API → 결과 이동/토스트. 빈 값 에러. |
| 목록/카드 | 클릭 → 상세 이동. 스크롤 → 추가 로딩(있는 경우). |
| 토글 | 클릭 → on/off 전환 → API → 반영. 재클릭 → 원복. |
| 삭제 | 삭제 → 확인 다이얼로그 → API → 목록에서 제거. |
| 모달/바텀시트 | 트리거 → 열림. 입력 → 제출 → 닫힘 + 반영. |
| 검색 | 키워드 입력 → 결과 표시. 빈 검색 → 안내. 결과 없음 → 빈 상태. |
| 탭/뷰 전환 | 탭 클릭 → 콘텐츠 변경. 전환 전 데이터 유지 여부. |
| 페이지네이션/무한스크롤 | 스크롤 → 추가 로딩. 마지막 페이지 → 로딩 중단. |
| 날짜 선택 | 날짜 클릭 → 해당 데이터 표시. 월/주 이동 → 데이터 갱신. |
| 지도 | 핀 표시. 핀 클릭 → 정보 표시. 영역 이동 → 핀 갱신. |
| 파일 업로드 | 파일 선택 → 미리보기. 업로드 → 성공/실패. 용량 초과 → 에러. |
| 드래그/정렬 | 드래그 → 순서 변경 → 저장. 새로고침 후 순서 유지. |

**완료 기준:**

- [ ] 브라우저에서 화면이 정상 렌더링됨
- [ ] 모든 인터랙션(클릭, 입력, 제출)이 동작함
- [ ] API 호출 후 결과가 화면에 반영됨
- [ ] 에러/로딩/빈 상태가 표시됨

## 2-3. 프론트↔백엔드 연동 확인

**절차:**

1. 프론트에서 실제 백엔드 API를 호출하여 데이터가 표시되는지 확인한다
2. MOCK 데이터나 하드코딩이 남아있지 않은지 확인한다
3. shared-types의 타입과 실제 API 응답이 일치하는지 확인한다

**완료 기준:**

- [ ] 프론트 화면에 실제 DB 데이터가 표시됨 (MOCK 아님)
- [ ] `grep -rn "MOCK_\|TODO\|FIXME" apps/web/` 결과 0건
- [ ] API 호출 경로(HTTP 메서드 + URL)가 백엔드 라우트와 일치
