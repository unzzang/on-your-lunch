# Phase 3. 리뷰 + 핸드오프

> PO 리뷰를 거쳐 개발팀에 전달한다.

---

## 3-1. PO 리뷰

**절차:**

1. 김지수가 전체 화면 디자인 완료를 보고한다
2. PO가 `design/` 폴더의 HTML 파일을 브라우저에서 확인한다
3. 아래 기준으로 리뷰한다
4. 피드백 → 김지수 반영 → 재리뷰

**리뷰 기준:**

| 항목 | 확인 내용 |
|------|----------|
| 기획 반영 | 기능 명세서(specs/)의 모든 기능이 디자인에 포함되었는가 |
| 화면 설계 반영 | 화면 설계(screen/)의 구조/인터랙션이 디자인에 반영되었는가 |
| 4가지 상태 | 모든 화면에 정상/빈/로딩/에러 상태가 있는가 |
| 디자인 원칙 | 8가지 원칙 준수 여부 (CTA 1개, 계층 3단계, 터치 44px 등) |
| 브랜드 일관성 | 디자인 시스템 토큰만 사용했는가 (임의 값 없음) |
| UX 흐름 | 핵심 플로우가 자연스러운가 (온보딩 → 홈 → 탐색 → 기록) |

**완료 기준:**

- [ ] PO 피드백 반영 완료
- [ ] PO 승인

---

## 3-2. 핸드오프

**절차:**

1. 디자인 파일 최종 정리 (`design/` 폴더)
2. 디자인 토큰 추출/확인 (`docs/005_design/design-system.md`)
3. 에셋 파일 정리 (`design/assets/`)
4. 개발팀에 전달

**핸드오프 산출물 체크리스트:**

| 산출물 | 위치 | 확인 |
|--------|------|------|
| 디자인 시스템 문서 | `docs/005_design/design-system.md` | - [ ] 컬러/타이포/간격/컴포넌트 스펙 포함 |
| 화면 디자인 HTML | `design/{화면명}.html` | - [ ] 모든 화면 포함 (4가지 상태) |
| 에셋 파일 | `design/assets/` | - [ ] 아이콘, 이미지 포함 |
| 디자인 토큰 값 | `docs/005_design/design-system.md` | - [ ] 개발에서 바로 사용 가능한 값 (HEX, px) |

**QA와의 연결:**

핸드오프된 `design/` 폴더의 HTML 파일은 **Stage 5 QA Phase 3-D (디자인 검증)** 에서 실제 구현과 비교하는 기준이 된다.

```
design/{화면명}.html (디자인 기준)
    ↕ 브라우저에서 비교
http://localhost:3001/{화면경로} (실제 구현)
```

**완료 기준:**

- [ ] 핸드오프 산출물 체크리스트 전체 체크
- [ ] 개발팀이 디자인 파일을 확인하고 질문 없음
- [ ] 개발 시작 가능 상태
