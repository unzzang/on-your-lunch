# 04. 배달 연동 및 기록

> 작성일: 2026-03-30
> 작성자: 도리
> 상태: 초안

## 개요

투표로 확정된 메뉴를 배달앱(배달의민족, 쿠팡이츠)에서 바로 주문할 수 있도록 딥링크로 연결하고, 확정된 점심 메뉴를 날짜/메뉴/날씨와 함께 자동 기록하여 히스토리를 관리하는 기능이다.

## 사용자 스토리

- 신입사원으로서, 투표 결과가 나오면 배달앱으로 바로 이동해 주문하고 싶다. 메뉴명을 다시 검색하는 번거로움을 없애기 위해서다.
- 신입사원으로서, 오늘 뭐 먹었는지 자동으로 기록되길 원한다. "저번주에 뭐 먹었지?" 질문에 바로 답하고, 같은 메뉴 반복을 피하기 위해서다.

## 기능 상세

### 4-1. 배달앱 딥링크

**동작:**

1. 투표 결과 화면에서 확정된 메뉴명 하단에 "배달의민족으로 주문" / "쿠팡이츠로 주문" 버튼 2개를 노출한다.
2. 버튼 탭 시 해당 배달앱의 메뉴 검색 화면으로 이동한다.
3. 딥링크 URL 스킴 구조:
   - **배달의민족:** `baemin://search?query={encodeURIComponent(메뉴명)}`
     - fallback (앱 미설치): `https://play.google.com/store/apps/details?id=com.sampleapp.baemin` (Android) / `https://apps.apple.com/kr/app/id378084485` (iOS)
   - **쿠팡이츠:** `coupangeats://search?keyword={encodeURIComponent(메뉴명)}`
     - fallback (앱 미설치): `https://play.google.com/store/apps/details?id=com.coupang.eats` (Android) / `https://apps.apple.com/kr/app/id1445504255` (iOS)
4. 딥링크 호출 후 500ms 내에 앱 전환이 발생하지 않으면 fallback URL(스토어)로 리다이렉트한다.
5. 딥링크 호출 시 React Native의 `Linking.canOpenURL()` → `Linking.openURL()` 순서로 처리한다.
   - `canOpenURL()`이 `false`를 반환하면 즉시 fallback URL로 이동한다.

**참고:** 배달의민족, 쿠팡이츠의 딥링크 URL 스킴은 공식 문서가 비공개이므로, 출시 전 실제 앱에서 검증이 필요하다. 위 스킴은 리버스 엔지니어링 기반 추정값이며, 정확한 스킴은 각 플랫폼 파트너십 문의 또는 출시 전 QA에서 확정한다.

**정책:**

- 딥링크 버튼은 투표 상태가 "확정(CONFIRMED)"인 경우에만 활성화된다. 투표 진행 중이거나 미확정 상태에서는 버튼을 비활성(disabled) 처리한다.
- **웹 참여자 배달 연동:** 웹 참여자는 배달앱 딥링크를 사용할 수 없으므로, 배달앱 모바일 웹 URL로 연결한다.
  - 배달의민족: `https://m.baemin.com/search?query={encodeURIComponent(메뉴명)}`
  - 쿠팡이츠: `https://m.coupangeats.com/search?keyword={encodeURIComponent(메뉴명)}`
- 메뉴명에 특수문자가 포함된 경우 `encodeURIComponent()`로 인코딩하여 전달한다.
- 딥링크 버튼 탭 이벤트를 로깅한다. 로깅 항목: `userId`, `menuName`, `targetApp("baemin" | "coupangeats")`, `timestamp`, `result("opened" | "fallback")`.
- iOS에서는 `Info.plist`의 `LSApplicationQueriesSchemes`에 `baemin`, `coupangeats` 스킴을 등록해야 `canOpenURL()`이 동작한다.
- Android에서는 Android 11(API 30) 이상 대응을 위해 `AndroidManifest.xml`의 `<queries>` 태그에 해당 패키지를 선언한다.

**에러 케이스:**

| 상황 | 처리 |
|------|------|
| 배달앱 미설치 | 해당 앱 스토어(Google Play / App Store) 페이지로 이동 |
| 딥링크 호출 실패 (OS 에러) | 토스트 메시지: "배달앱 연결에 실패했어요. 직접 앱을 열어주세요." |
| 네트워크 미연결 상태에서 fallback URL 접근 불가 | 토스트 메시지: "인터넷 연결을 확인해주세요." |
| 메뉴명이 빈 문자열 | 딥링크 버튼 비활성화. 발생 시 에러 로깅 |

### 5-1. 오늘의 점심 기록

**동작:**

1. 점심 기록 생성 트리거 (다음 중 하나):
   - **투표 확정 시:** 투표 결과가 "확정(CONFIRMED)" 상태로 전환되는 시점에 서버가 자동으로 점심 기록을 생성한다.
   - **혼자 주문하기 시:** 사용자가 AI 추천 메뉴에서 "혼자 주문하기"를 선택하면 투표 없이 점심 기록을 생성한다. 이 경우 `groupId = null`, `voteId = null`이다.
2. 저장 데이터 항목:
   - `id`: UUID, PK
   - `groupId`: 투표방 ID를 그룹 식별자로 사용. 별도 그룹 관리 없음. 혼자 주문 시 `null`
   - `voteId`: 해당 투표 ID
   - `menuName`: 확정된 메뉴명 (문자열, 최대 50자)
   - `category`: 메뉴 카테고리 (한식, 중식, 일식, 양식, 분식, 치킨/피자, 버거, 아시안, 샐러드, 도시락 — 장르 마스터 10개)
   - `weather`: 기록 시점의 날씨 정보 (맑음/흐림/비/눈/기타) — AI 추천 시 사용된 날씨 데이터를 그대로 저장
   - `temperature`: 기록 시점의 기온 (섭씨, 정수)
   - `participantCount`: 투표 참여 인원 수
   - `date`: 기록 날짜 (YYYY-MM-DD, 한국 시간 기준)
   - `createdAt`: 레코드 생성 시각 (ISO 8601)
3. 사용자는 "점심 기록" 탭에서 자신이 속한 그룹의 기록을 날짜 역순으로 조회한다.
4. 기록 목록 화면: 날짜, 메뉴명, 날씨 아이콘, 카테고리 태그를 카드 형태로 표시한다.
5. 기록 상세 화면: 해당 투표의 전체 후보 메뉴, 각 메뉴 득표수, 최종 확정 메뉴, 참여 인원을 표시한다.

**정책:**

- 같은 투표방에서 하루 1건만 생성된다. 동일 투표방에서 같은 날짜에 투표가 여러 번 확정되면 마지막 확정 결과로 덮어쓴다. 혼자 주문 기록(`groupId = null`)은 별도 1건 허용된다.
- 기록은 삭제 불가. 수정 불가. 읽기 전용 데이터이다.
- 기록 조회 API는 페이지네이션을 적용한다. 1페이지당 20건, cursor 기반 페이지네이션.
- 날씨 데이터가 없는 경우 (API 장애 등) `weather` 필드는 `null`로 저장하고, UI에서는 날씨 아이콘 대신 "-" 표시.
- 기록 보관 기간: 제한 없음 (서비스 이용 기간 동안 영구 보관).

**에러 케이스:**

| 상황 | 처리 |
|------|------|
| 투표 확정 시점에 서버 장애로 기록 생성 실패 | 실패 이벤트를 큐에 저장하고, 서버 복구 후 재시도 (최대 3회, 간격 1분/5분/30분) |
| 날씨 API 응답 실패 | `weather`, `temperature` 필드를 `null`로 저장. 기록 자체는 정상 생성 |
| 동일 날짜 + 동일 투표방 중복 기록 요청 | 기존 기록을 UPDATE (UPSERT). 신규 INSERT 하지 않음 |
| 투표방에 속하지 않은 사용자가 기록 조회 시도 | HTTP 403 응답. 클라이언트에서 "접근 권한이 없어요." 메시지 표시 |
