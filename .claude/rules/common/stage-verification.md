# Stage 검증 규칙

모든 Stage는 완료 전 검증을 거쳐야 한다. 검증 없이 다음 Stage로 진입할 수 없다.

## 검증 흐름

```
담당 에이전트 작업 완료
    ↓
[1차] 전문 검증 (해당 Stage에 있을 때만)
    ↓ 통과
[2차] 정합성 검증 (송현아 또는 진도준)
    ↓ 통과/조건부 통과
[보고] PO에게 검증 결과 보고 (텔레그램 알림)
    ↓
[최종] PO 승인 (또는 위더 대리 승인)
    ↓
Stage 완료 → 다음 Stage 진입
```

검증 실패 시: 담당 에이전트에게 수정 사항 전달 → 수정 → 재검증. **PO 개입 없이** 통과할 때까지 반복한다. PO에게 보고하는 것은 **통과 또는 조건부 통과** 판정 시에만.

## 판정 기준

### 이슈 심각도 정의

| 심각도 | 정의 | 예시 |
|--------|------|------|
| **상** | 다음 Stage 작업이 불가능하거나, 사용자에게 직접 피해 | 접근성 위반, 핵심 산출물 누락, 빌드 실패 |
| **중** | 다음 Stage 작업에 혼란을 주거나, 품질 저하 | 문서 불일치, 하위 화면 미구현, 토큰 미정의 |
| **하** | 개선하면 좋으나 다음 Stage 진행에 지장 없음 | 토큰 미세 불일치, 정렬, 코드 스타일 |

### 판정 기준

| 판정 | 조건 | 다음 Stage 진입 | PO 보고 |
|------|------|:-:|:-:|
| **통과** | 상 0건 + 중 0건 | O | O (승인 요청) |
| **조건부 통과** | 상 0건 + 중 3건 이하 | O (병행 수정) | O (승인 요청) |
| **실패** | 상 1건 이상, 또는 중 4건 이상 | X (수정 후 재검증) | X (수정 완료까지 대기) |

- **하** 이슈는 판정에 영향 없음. 권장 수정으로 기록만 한다.
- **조건부 통과** 시: 다음 Stage 진행 가능하나, 중 이슈는 해당 Stage 내에서 병행 수정한다.

## Stage별 검증 담당

| Stage | 1차 검증 (전문) | 2차 검증 (정합성) | 최종 승인 |
|-------|----------------|-----------------|----------|
| 0. 셋업 | - | 송현아 (산출물 검증) | PO/위더 |
| 1. 사전조사 | - | 송현아 (출처·정합성) | PO/위더 |
| 2. 기획 | 이명환 (기술 스펙 타당성) | 송현아 (명세 간 정합성) | PO/위더 |
| 3. 디자인 | 김지수 (디자인 원칙 준수) | 송현아 (토큰·체크리스트) | PO/위더 |
| 4. 개발 | 이명환 (코드 리뷰) | 진도준 (빌드·테스트·린트) | PO/위더 |
| 5. QA | - | 진도준 (QA 4단계) | PO/위더 |
| 6. 출시 | - | 송현아 + 진도준 | PO/위더 |

## 검증 기준

### 송현아 (산출물 정합성 검증)
- 문서 간 불일치 없는가 (명세↔화면↔API↔ERD)
- 가짜 데이터·모호 표현이 없는가
- 모든 수치에 출처가 있는가
- 누락된 항목이 없는가

### 진도준 (QA 검증)
- 빌드·테스트·린트 전체 통과하는가
- shared-types와 실제 API 응답이 일치하는가
- 기능 명세서의 에러 케이스가 구현되었는가
- 시뮬레이터에서 기본 동작이 확인되는가

### 이명환 (기술 검증)
- 기술 스펙이 기능 명세를 모두 커버하는가
- API 설계가 RESTful 원칙에 맞는가
- ERD 인덱스와 제약조건이 적절한가
- 코드 아키텍처가 설계 문서를 따르는가

### 김지수 (디자인 검증)
- 디자인 시스템 토큰만 사용했는가 (임의 값 없음)
- 디자인 원칙 8가지를 준수했는가
- component-spec.md 체크리스트를 통과하는가
- 모든 상태(정상·로딩·빈·에러)가 정의되었는가

## 검증 결과 보고 형식

검증 완료 후 아래 형식으로 PO에게 보고한다:

```
[Stage N 검증 완료]

검증 대상: {Stage명}
1차 검증: {담당자} — {통과/실패} ({사유})
2차 검증: {담당자} — {통과/실패} ({사유})

이슈 요약: 상 N건 / 중 N건 / 하 N건

발견 사항:
- (있으면 기재)

판정: 통과 / 조건부 통과 / 실패
PO 승인 필요
```
