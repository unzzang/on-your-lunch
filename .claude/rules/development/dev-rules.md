# 개발 규칙

## 개발 계획 먼저 (절대 규칙)

**코드를 한 줄이라도 작성하기 전에 개발 계획서를 작성한다.** 계획 없이 코드부터 작성하지 않는다.

개발 계획서에 포함할 항목:
1. **프로젝트 구조** — 폴더/파일 트리 전체
2. **기술 스택** — 프레임워크, 라이브러리, 버전
3. **Phase별 작업 목록** — 각 Phase에서 생성할 파일과 내용
4. **파일 간 연결 관계** — 어떤 파일이 어떤 파일을 import하는지
5. **디자인 대조 기준** — 각 화면에서 측정할 항목
6. **삽질 방지 체크리스트** — 이전 개발에서 배운 교훈 반영

계획서를 PO에게 공유하고 승인 후 구현을 시작한다.

## 영역 침범 금지 (절대 규칙)

**프론트엔드 작업 시 `apps/api/` 폴더를 건드리지 않는다.** 백엔드 작업이 필요하면 즉시 멈추고 PO에게 보고한다.

```
❌ "빠르게 해결하려고" 백엔드 엔드포인트를 직접 만든다
❌ "간단한 수정이라" app.module.ts에 모듈을 추가한다
❌ "타입만 맞추려고" restaurant.service.ts를 수정한다
✅ "백엔드 엔드포인트가 필요합니다. 백엔드팀에 요청해주세요" → 멈춘다
```

이 규칙을 어기면 빌드가 깨지고 복구에 더 오래 걸린다. 2026-04-05에 지도 기능 추가하다 백엔드 dist/ 삭제 → 빌드 실패 → 전체 프로젝트 백업 후 재구축하는 결과를 낳았다.

## API 계약 먼저

코드를 작성하기 전에 프론트↔백엔드가 요청/응답 타입을 합의한다. shared-types에 정의.

## 기능 단위 개발

기능 명세서 1개 = 개발 단위 1개. 백엔드 API → 프론트 화면 → 연동 확인.

## Definition of Done

**백엔드와 프론트엔드 모두 완료되어야 Stage 4(개발) 완료로 판정한다.** 한쪽만 완료된 상태에서 QA 또는 다음 Stage로 넘기지 않는다.

아래를 **모두** 충족해야 QA에 넘긴다:

- 백엔드: API 비즈니스 로직 구현 완료 + 빌드/테스트/린트 통과
- 프론트엔드: **실제 화면 UI 구현 완료** (빈 컴포넌트/구조만은 미완료 상태)
- 프론트↔백엔드 연동 확인 (API 호출 → 화면 표시)
- **MOCK/하드코딩 데이터 전량 제거** — `grep -rn "MOCK_" app/` 결과 0건
- **API 훅 1:1 대조** — 백엔드 API 목록과 프론트 훅 목록을 대조하여 누락 0건
- shared-types와 실제 API 응답 일치
- 시뮬레이터에서 **모든 화면** 실제 데이터 표시 확인 (홈만이 아니라 전체)

## 개발 완료 필수 증빙

개발 완료를 보고할 때 아래 4가지를 반드시 첨부한다. 하나라도 없으면 미완료로 반려.

1. **빌드 로그** — 타입 체크/빌드/린트 에러 0건 증빙
2. **MOCK 검색 결과** — `grep -rn "MOCK_" app/` 결과 0건 증빙
3. **모든 화면 시뮬레이터 스크린샷** — 화면 수만큼 (홈만이 아니라 전체)
4. **핵심 플로우 테스트 결과** — 각 플로우 통과/실패 기록

## 디자인 대조 검수

화면 하나를 구현할 때마다 디자인 명세와 대조한다. **코드 수치만 비교하는 것은 검수가 아니다.**

### 검수 방법 (필수)

디자인이 HTML로 제공된 경우, 반드시 **브라우저에서 열고 inspect로 computed CSS를 측정**한다.

```bash
# 1. 디자인 HTML을 로컬 서버로 띄운다
cd design/ && python3 -m http.server 8899

# 2. browse 스킬로 열고 요소별 실제 렌더링 값을 측정한다
$B goto "http://localhost:8899/home.html"
$B inspect ".card"           # 카드 전체 dimensions
$B inspect ".card__name"     # 폰트 크기, 굵기, line-height, 렌더링 높이
$B inspect ".card__fav"      # 음수 마진 등 트릭 확인
$B css ".phone" "background-color"  # 배경색 확인
```

### 필수 측정 항목

| 항목 | 확인할 것 | 흔한 실수 |
|------|----------|----------|
| **배경색** | 페이지/섹션별 배경색 구분 | 전부 흰색으로 통일 |
| **font-size** | 요소별 실제 크기 | 디자인 토큰 값과 HTML 값이 다를 수 있음 |
| **line-height** | computed 높이 측정 | 토큰의 lineHeight를 단일행 텍스트에 적용하면 카드가 커짐 |
| **margin (음수 포함)** | 버튼/아이콘의 음수 마진 | 터치 영역(44px) 버튼이 카드 높이를 키움 |
| **font-weight** | 700 vs 600 구분 | typo 토큰이 600인데 디자인이 700인 경우 |
| **이미지/에셋** | 디자인에 이미지가 있으면 코드에도 반드시 포함 | placeholder 아이콘으로 대체 금지 |

### 체크리스트

- [ ] 브라우저에서 디자인 HTML을 열고 inspect로 측정했는가 (코드만 읽은 건 불가)
- [ ] 배경색이 섹션별로 디자인과 일치하는가 (bg-primary vs bg-secondary 구분)
- [ ] 카드/리스트 아이템의 전체 높이(dimensions)가 디자인과 일치하는가
- [ ] 텍스트의 font-size, font-weight, line-height가 computed 값과 일치하는가
- [ ] 디자인에 있는 이미지가 코드에도 표시되는가 (테스트 이미지라도 넣을 것)
- [ ] 아이콘이 디자인과 같은 라이브러리/스타일인가 (이모지 대체 금지)
- [ ] 4가지 상태(정상/로딩/빈/에러)가 디자인에 맞게 구현되었는가
- [ ] 시뮬레이터에서 실제 화면을 확인했는가

디자인과 불일치하는 부분이 있으면 즉시 수정 후 다음 화면으로 진행한다.

자체 테스트 통과 ≠ 수정 완료. QA 엔지니어의 재검증(Verified) 판정까지 "수정 중" 상태.

## 프로젝트 세팅 규칙

프로젝트 초기 세팅 시 반드시 수행:

1. **CLAUDE.md에 프로젝트 구조 반영** — 실제 생성된 폴더/파일 구조를 CLAUDE.md의 "프로젝트 구조" 섹션에 기록한다. 개발팀이 작업 시작 시 CLAUDE.md만 보고 전체 구조를 파악할 수 있어야 한다.
2. **CLAUDE.md에 개발 명령어 반영** — 빌드, 테스트, 린트, 마이그레이션 등 실행 명령어를 기록한다.
3. **shared-types 먼저 정의** — 프론트↔백엔드 API 타입을 shared-types에 먼저 합의한 후 구현에 착수한다.

## node_modules 규칙

- **워크스페이스 루트(`src/`)에만 node_modules를 설치한다.** 하위 패키지(`apps/api/`, `apps/mobile/`, `packages/shared-types/`)에 개별 node_modules를 생성하지 않는다.
- pnpm workspace가 루트의 node_modules를 공유하는 구조이므로, 하위에서 `npm install`이나 `pnpm install`을 개별 실행하지 않는다.
- 의존성 추가 시: 루트에서 `pnpm add {패키지} --filter {워크스페이스명}` 으로 설치한다.
- 하위 패키지에 node_modules가 발견되면 즉시 삭제하고 루트에서 `pnpm install`을 다시 실행한다.

## 프론트엔드 개발 규칙 (시뮬레이터 필수)

프론트엔드 개발 시 **시뮬레이터를 먼저 띄워놓고, 화면 하나 만들 때마다 확인**한다. 파일을 한꺼번에 생성하고 마지막에 실행하면 에러가 폭발한다.

### 올바른 순서

```
1. 시뮬레이터 먼저 실행 (npx expo start --ios --offline)
2. _layout.tsx 최소 버전 생성 → 시뮬레이터 확인 (빈 화면 뜨는지)
3. 탭 레이아웃 추가 → 시뮬레이터 확인 (탭 바 뜨는지)
4. 홈 화면 UI 추가 → 시뮬레이터 확인
5. 한 화면씩 추가 → 매번 시뮬레이터 확인
```

### 절대 하지 말 것

```
❌ 에이전트에게 "화면 20개 한꺼번에 만들어줘" → 마지막에 실행 → 에러 폭발
✅ "화면 1개 만들고 시뮬레이터 확인" → "다음 화면" → 반복
```

### ScrollView 규칙 (스크롤 안 되는 버그 방지)

- `react-native-gesture-handler`의 `ScrollView` 사용 금지. `GestureHandlerRootView`와 제스처 충돌으로 스크롤이 동작하지 않음.
- 반드시 `react-native`의 `ScrollView` 또는 `FlatList`를 사용한다.
- 리스트 형태의 화면은 `FlatList` + `ListHeaderComponent` 패턴 권장.

### pnpm workspace + Expo 필수 설정 (누락 시 앱 크래시)

- **`metro.config.js` 필수** — pnpm workspace에서 node_modules가 루트에 hoisted되므로, Metro Bundler에게 workspace 루트를 알려줘야 함. 이 파일이 없으면 의존성 해석 실패로 앱이 크래시.
- **`babel.config.js`** — `@/` 경로 별칭 사용 시 `babel-plugin-module-resolver` 필요.

### expo-router 필수 파일 (누락 시 크래시)

- `app/index.tsx` — 루트 진입점. 없으면 expo-router가 앱을 시작 못 함.
- `app/_layout.tsx` — 루트 레이아웃. Provider 등록.
- 괄호 그룹(`(auth)/`, `(tabs)/`)에는 반드시 `_layout.tsx` 필요.
- `<Stack>` 안에는 `<Stack.Screen>`만. `<StatusBar>` 같은 건 Stack 바깥에.
- app.json에서 참조하는 asset 파일(icon.png 등)이 실제로 있어야 함.

### 탭 네비게이션 규칙: 각 탭은 독립 스택

각 탭에서 하위 화면(식당 상세 등)으로 이동한 뒤 **뒤로 가기를 누르면 같은 탭의 이전 화면**으로 돌아가야 한다. 다른 탭으로 빠지면 안 된다.

```
✅ 올바른 동작:
  홈 탭 → 식당 상세 → 뒤로 → 홈 탭
  탐색 탭 → 식당 상세 → 뒤로 → 탐색 탭
  마이 탭 → 프로필 편집 → 뒤로 → 마이 탭

❌ 잘못된 동작:
  홈 탭 → 식당 상세 → 뒤로 → 탐색 탭 (다른 탭으로 빠짐)
```

이를 위해 각 탭 폴더 안에 하위 화면을 넣거나, expo-router의 `router.push()` 대신 탭 내부 Stack을 사용해야 한다.
