# 기능 심층 분석 (Deep Dive)

특정 앱 서비스의 특정 기능 하나를 골라, 겉으로 보이는 UI부터 엣지 케이스 대응까지 낱낱이 파헤치는 분석이다.

`/app-analysis`가 앱 전체를 넓게 보는 거라면, `/deep-dive`는 **기능 하나를 깊게 판다.**

---

## 실행 조건

- PO 또는 위더가 분석 대상(서비스명 + 기능명)을 지정
- 예시: "배민의 온보딩", "토스의 송금 에러 처리", "당근의 채팅 신고 기능"

---

## 4단계 프로세스

```
Phase 0: 분석 설계       — 위더 (대상, 범위, 핵심 질문 정의)
Phase 1: 데이터 수집     — 앱분석전문가 (웹 검색 + PO 제공 스크린샷)
Phase 2: 심층 분석       — 앱분석전문가 (7개 분석 레이어)
Phase 3: 시사점 정리     — 앱분석전문가 (발견 + 적용 포인트)
```

---

## Phase 0: 분석 설계

위더가 PO와 함께 아래를 확정한다.

| 항목 | 내용 |
|------|------|
| **서비스명** | 분석 대상 앱 (예: 배달의민족) |
| **기능명** | 분석할 특정 기능 (예: 온보딩, 고객센터, 리뷰 작성) |
| **분석 목적** | 왜 이 기능을 분석하는가 (벤치마크 / 우리 서비스 참고 / 개선 아이디어) |
| **핵심 질문 (KRQ)** | 이 분석을 통해 답해야 할 질문 3~5개 |
| **PO 스크린샷** | PO가 직접 앱을 사용하며 캡처한 스크린샷 (있으면 첨부) |

### PO 스크린샷 가이드

PO가 스크린샷을 제공할 경우 아래를 포함하면 분석 품질이 올라간다:

- 기능의 **진입점** (어디서 이 기능에 접근하는지)
- **정상 플로우** 전체 (각 단계별 1장씩)
- **예외 상황** (에러, 빈 상태, 권한 요청 등)
- **안내 문구**가 보이는 화면 (팝업, 토스트, 모달, 툴팁)
- 캡처 시 **날짜**와 **앱 버전** 기록

---

## Phase 1: 데이터 수집

앱분석전문가가 아래 소스에서 데이터를 수집한다.

| 소스 | 수집 항목 |
|------|----------|
| **PO 스크린샷** | 제공된 스크린샷 분석 (가장 신뢰도 높은 소스) |
| **웹 검색** | 해당 기능 관련 UX 분석 글, 블로그, 사용기 |
| **앱스토어 리뷰** | 해당 기능 언급 리뷰 (긍정/부정) |
| **공식 가이드/FAQ** | 해당 기능 관련 고객센터 문서, 도움말 |
| **커뮤니티** | 사용자 반응, 불만, 팁 공유 |
| **업데이트 이력** | 해당 기능의 변경 이력 (릴리스 노트에서 추출) |

### 수집 규칙

- 모든 데이터에 **수집 날짜**와 **출처 URL** 기록
- PO 스크린샷은 출처를 "PO 직접 캡처 ({날짜})"로 표기
- 수집 불가 항목은 "데이터 없음 — 수집 방법: {제안}" 표기
- AI가 추정/생성한 수치 사용 절대 금지

## Phase 2: 심층 분석 — 7개 레이어

하나의 기능을 7개 관점으로 분해한다. **모든 레이어를 빠짐없이 분석한다.**

### Layer 1. 플로우 해부 (Flow Anatomy)

기능의 전체 흐름을 단계별로 분해한다.

| 분석 항목 | 세부 |
|----------|------|
| **진입 경로** | 이 기능에 도달하는 모든 경로 (탭, 딥링크, 푸시, 배너 등) |
| **단계별 화면** | 각 단계에서 보이는 화면, 순서, 필수/선택 구분 |
| **분기 조건** | 어떤 조건에서 다른 경로로 빠지는지 (로그인 여부, 권한, 데이터 유무) |
| **완료/이탈 지점** | 성공 완료 화면, 중간 이탈 시 상태 보존 여부 |
| **되돌아가기** | 뒤로 가기 시 동작, 데이터 보존, 확인 팝업 유무 |

### Layer 2. 마이크로카피 수집 (Microcopy Inventory)

기능 내 모든 텍스트를 수집하고 분류한다.

| 분류 | 예시 |
|------|------|
| **제목/헤딩** | 화면 타이틀, 섹션 타이틀 |
| **본문/설명** | 기능 설명, 안내 문구 |
| **CTA 버튼** | 버튼 텍스트 (동사형? 명사형? 톤?) |
| **플레이스홀더** | 입력 필드 안내 텍스트 |
| **토스트/스낵바** | 완료/실패/경고 알림 문구 |
| **모달/팝업** | 확인 요청, 경고, 안내 팝업 문구 |
| **툴팁/코치마크** | 부가 설명, 첫 사용 안내 |
| **에러 메시지** | 입력 오류, 네트워크 오류, 서버 오류별 문구 |
| **빈 상태 문구** | 데이터가 없을 때 안내 |

각 문구에 대해:
- **원문 그대로** 기록 (인용)
- **톤** 판단 (존댓말/반말, 친근/격식, 이모지 사용 여부)
- **의도** 해석 (안내? 유도? 경고? 위로?)

### Layer 3. 상태별 대응 (State Handling)

기능이 처리하는 모든 상태와 그 대응을 기록한다.

| 상태 | 확인 항목 |
|------|----------|
| **정상 상태** | 데이터가 있고 모든 조건이 충족된 화면 |
| **로딩 상태** | 스켈레톤? 스피너? 프로그레스바? 없음? |
| **빈 상태** | 데이터 0건일 때 — 문구, 일러스트, CTA 유무 |
| **에러 상태** | 네트워크 끊김, 서버 오류, 타임아웃 — 각각 다른 대응인지 |
| **권한 부족** | 로그인 필요, 위치 권한, 알림 권한 — 어떻게 요청하는지 |
| **제한/차단** | 횟수 제한, 시간 제한, 차단된 사용자 — 안내 방식 |
| **최초 사용** | 온보딩, 코치마크, 첫 사용 가이드 유무 |
| **재방문** | 이전 상태 복원, 최근 항목 노출 여부 |

### Layer 4. 엣지 케이스 & 예외 처리 (Edge Cases)

일반적이지 않은 상황에서의 대응을 분석한다.

| 상황 | 확인 항목 |
|------|----------|
| **입력 극단값** | 매우 긴 텍스트, 특수문자, 이모지 입력 시 처리 |
| **동시 동작** | 같은 기능을 두 번 실행, 빠른 더블 탭 |
| **중간 이탈** | 앱 종료 후 재진입 시 상태, 백그라운드 복귀 |
| **네트워크 전환** | Wi-Fi → LTE, 비행기 모드 해제 시 동작 |
| **데이터 경계** | 0건, 1건, 대량(100건+) 데이터 시 UI 변화 |
| **버전/OS 차이** | iOS vs Android 차이, 구형 기기 대응 |
| **다국어** | 한국어 외 언어 지원 여부, 긴 번역 시 UI 깨짐 |

### Layer 5. 인터랙션 패턴 (Interaction Patterns)

사용자가 기능을 사용할 때의 조작 방식을 분석한다.

| 항목 | 세부 |
|------|------|
| **제스처** | 탭, 스와이프, 롱프레스, 핀치, 드래그 — 어디서 어떤 제스처를 쓰는지 |
| **피드백** | 탭 시 시각/촉각/사운드 피드백 유무 |
| **전환 애니메이션** | 화면 간 전환 방식 (push, modal, fade, none) |
| **스크롤 동작** | 무한 스크롤 vs 페이지네이션, pull-to-refresh 유무 |
| **입력 보조** | 자동완성, 음성 입력, 바코드 스캔 등 입력 편의 기능 |
| **단축 동작** | 반복 작업을 줄이는 UI (최근 항목, 즐겨찾기, 자동 저장) |

### Layer 6. 비즈니스 로직 추론 (Business Logic)

기능 뒤에 숨겨진 비즈니스 규칙을 추론한다.

| 항목 | 세부 |
|------|------|
| **정책 규칙** | 제한 횟수, 유효 기간, 자격 조건 등 |
| **우선순위 로직** | 리스트 정렬 기준, 추천 알고리즘 단서 |
| **과금 포인트** | 무료/유료 경계, 프리미엄 유도 방식 |
| **법적/규제** | 약관 동의, 개인정보 처리, 미성년자 제한 |
| **A/B 테스트 흔적** | 다른 사용자와 다른 UI가 보이는 경우 |

### Layer 7. 접근성 & 포용성 (Accessibility)

다양한 사용자 환경에서의 대응을 확인한다.

| 항목 | 세부 |
|------|------|
| **VoiceOver/TalkBack** | 스크린 리더 지원 여부, 읽기 순서 |
| **다크 모드** | 지원 여부, 대비 유지 여부 |
| **큰 글씨** | 시스템 글자 크기 변경 시 UI 대응 |
| **한 손 조작** | 주요 동작이 엄지 영역 내에 있는지 |

## Phase 3: 시사점 정리

### "So What?" 규칙

모든 분석 항목에 아래 구조를 강제한다:

```
[사실] {데이터/관찰}
[해석] {이것이 의미하는 바}
[적용] {우리 서비스에 가져갈 수 있는 것}
```

데이터만 나열하고 해석이 없는 분석은 미완성으로 판정한다.

### 최종 보고서 구조

```markdown
# {서비스명} — {기능명} 심층 분석

## 분석 개요
- 서비스: {서비스명}
- 기능: {기능명}
- 분석 일자: {YYYY-MM-DD}
- 분석 목적: {목적}
- 핵심 질문(KRQ): {질문 목록}

## 핵심 발견 (3~5줄 요약)

## KRQ별 답변
- KRQ 1: {질문} → {답변 + 근거}
- ...

## 7개 레이어 분석 결과
### 1. 플로우 해부
### 2. 마이크로카피 수집
### 3. 상태별 대응
### 4. 엣지 케이스 & 예외 처리
### 5. 인터랙션 패턴
### 6. 비즈니스 로직 추론
### 7. 접근성 & 포용성

## 잘한 점 (배울 것)
- {구체적 사례 + 근거}

## 아쉬운 점 (피할 것)
- {구체적 사례 + 근거}

## 우리 서비스 적용 포인트
- Must-Have: {필수 적용}
- Nice-to-Have: {선택 적용}
- Avoid: {하지 말 것}

## 데이터 출처 목록
- 출처 URL + 수집 날짜
```

## 산출물

| 산출물 | 저장 위치 |
|--------|----------|
| 분석 설계 | `.documents/deep-dive/{서비스명}/` |
| 심층 분석 보고서 | `.documents/deep-dive/{서비스명}/{YYYY-MM-DD}_{기능명}_분석.md` |

### 파일명 규칙

- 서비스명: 한글 그대로 사용 (예: `배달의민족`, `토스`, `당근마켓`)
- 기능명: 한글 그대로 사용 (예: `온보딩`, `고객센터`, `송금_에러처리`)
- 띄어쓰기는 언더스코어(`_`)로 대체

### 예시

```
.documents/deep-dive/
├── 배달의민족/
│   ├── 2026-04-05_온보딩_분석.md
│   ├── 2026-04-10_고객센터_분석.md
│   └── 2026-04-15_리뷰_작성_분석.md
└── 토스/
    └── 2026-04-07_송금_에러처리_분석.md
```

## 실행 규칙

- 위더는 Phase 0만 수행하고, Phase 1~3은 앱분석전문가에게 위임한다
- PO가 스크린샷을 제공하면 이를 1순위 소스로 활용한다
- 스크린샷 없이도 웹 검색 기반으로 분석 가능하되, 정확도 한계를 명시한다
- 가짜 데이터 절대 금지 — 확인 불가 항목은 "확인 불가 — {사유}" 표기
- 7개 레이어 중 확인 불가한 항목은 건너뛰되, "미확인" 표기 필수
- 각 Phase는 순차 진행한다

## `/app-analysis`와의 관계

| 구분 | `/app-analysis` | `/deep-dive` |
|------|----------------|-------------|
| **범위** | 앱 전체 (4영역 분석) | 특정 기능 1개 (7레이어 분석) |
| **깊이** | 넓고 전략적 | 좁고 디테일 |
| **산출물** | 종합 보고서 + 영역별 보고서 | 기능별 분석 문서 |
| **활용** | 시장/경쟁 파악, 진입 전략 | 구현 참고, UX 벤치마크, 마이크로카피 |
| **저장** | `.documents/app-analysis/` 또는 프로젝트 `.docs/` | `.documents/deep-dive/` |

필요 시 `/app-analysis`에서 특정 기능이 흥미로우면 `/deep-dive`로 이어서 파볼 수 있다.
