# 개발 프로세스 (Stage 4)

> 적용 범위: 아키텍처 결정부터 QA 투입 준비까지
> 책임자: PO (의사결정), 개발팀 (구현)
> 선행 조건: Stage 2 Gate 3 통과 + Stage 3 Phase 4 완료

---

## 1. 역할

확정된 설계와 디자인을 기반으로 동작하는 소프트웨어를 구현한다.

## 2. 프로세스

```
Phase 1 → Phase 2 → Phase 3 → Phase 4 → Phase 5
```

### Phase 1. 아키텍처 결정

- `references/architecture-patterns.md` 참조하여 패턴 조합 선정
- 이명환이 상세 아키텍처 확정 → DR 기록
- `CLAUDE.md`에 기술 스택 + 코드 아키텍처 반영

### Phase 2. 환경 셋업

- 모노레포/폴더 구조 생성
- 패키지 매니저, 빌드 도구, 린터 설정
- 개발/스테이징/프로덕션 환경 분리

### Phase 3. API 계약 합의

- `shared-types/`를 단일 소스로 사용
- 백엔드(이인수) + 프론트엔드(잭도슨) 합의
- 엔드포인트별 요청/응답 타입 정의

### Phase 4. 기능 단위 개발

- 기능 명세서 1개 = 개발 단위 1개
- 백엔드 → 프론트엔드 순서 또는 병렬 (API 계약 기반)
- 기능 완료 시 자체 테스트 포함

### Phase 5. QA 투입 준비

기능 단위별 Definition of Done:

- [ ] 빌드 성공
- [ ] 단위 테스트 통과
- [ ] API 계약 일치
- [ ] 린트/타입 체크 통과
- [ ] 기능 명세서 대비 동작 확인

## 3. 완료 기준

- [ ] 아키텍처 DR 기록
- [ ] MVP 전체 기능 구현 완료
- [ ] 모든 기능이 Definition of Done 충족
- [ ] QA 투입 가능 상태
