# QA/검증 규칙

## QA 최종 판정권

QA 엔지니어(진도준)가 QA 통과 판정권을 가진다. QA 통과 없이 출시하지 않는다.

## QA 4단계 (건너뛰기 금지)

```
Phase 1.   정적 검증     — 빌드, 테스트, 린트, 타입 체크
Phase 2.   통합 검증     — 2-1. 스키마↔백엔드, 2-2. API 계약↔코드, 환경 정합성
Phase 3-A. API 기능 검증  — 전체 엔드포인트 curl 테스트 + 에러 케이스
Phase 3-B. UI 인터랙션   — 브라우저에서 18개 기능 시나리오 직접 조작 확인 (필수)
Phase 3-C. 에러 시나리오  — 네트워크 에러, 빈 상태, 미인증 리다이렉트
Phase 3-D. 디자인 검증   — 디자인팀(김지수)이 디자인 HTML ↔ 실제 구현 브라우저 대조
Phase 4.   수정 재검증    — 버그 재현 확인, 리그레션
```

## 버그 상태 관리

Open → In Progress → Fixed → Verified → Closed

## 검증전문가 역할 (송현아)

- 기획 산출물 정합성 검증 (명세 간 불일치, 모호 표현)
- 콘텐츠 팩트 체크 (블로그, 영상 스크립트)
- Stage Gate 판정 의견 제시

## 기획 단계부터 참여

기능 맵 완성 후 QA 리뷰. 개발 후 QA보다 비용이 훨씬 저렴.
