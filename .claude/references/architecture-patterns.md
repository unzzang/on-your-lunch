# 소프트웨어 아키텍처 패턴 카탈로그

> 이 문서는 프로젝트 독립적인 참조 문서이다.
> 프로젝트 시작 시 이 카탈로그에서 적합한 패턴을 선택하고, 의사결정 기록(DR)에 근거를 남긴다.

---

## 사용법

```
1. 프로젝트 요구사항 확인 (플랫폼, 팀 규모, 서비스 복잡도)
2. 이 문서에서 3가지 카테고리별로 패턴 선택:
   ├─ A. 프로젝트 구조 패턴 (코드를 어떻게 관리할 건가?)
   ├─ B. 백엔드 아키텍처 패턴 (서버를 어떻게 구성할 건가?)
   └─ C. 프론트엔드 아키텍처 패턴 (클라이언트를 어떻게 구성할 건가?)
3. 선택한 패턴 조합으로 폴더 구조 확정
4. DR에 선택 근거 기록
```

---

## A. 프로젝트 구조 패턴

### A-1. 문서-코드 분리 모노레포

| 항목            | 내용                                                                                                       |
| --------------- | ---------------------------------------------------------------------------------------------------------- |
| **적합한 상황** | 모바일+웹+API 서비스, 1~5명 팀, 공유 타입 필요                                                             |
| **장점**        | 문서와 코드가 명확히 분리됨. 코드 내부는 모노레포 이점 유지. node_modules가 src/ 안에 격리되어 루트가 깔끔 |
| **단점**        | src/ 안에서의 의존성 관리는 여전히 필요                                                                    |
| **빌드 도구**   | pnpm workspace + Turborepo / Nx                                                                            |

### A-2. 순수 모노레포

| 항목            | 내용                                                  |
| --------------- | ----------------------------------------------------- |
| **적합한 상황** | 개발 중심 프로젝트, 문서가 적은 경우                  |
| **장점**        | 업계 표준 구조. 대부분의 도구/튜토리얼이 이 구조 가정 |
| **단점**        | 문서, node_modules, 설정 파일이 루트에 혼재           |

### A-3. 폴리레포

| 항목            | 내용                                                  |
| --------------- | ----------------------------------------------------- |
| **적합한 상황** | 팀이 크고 (10명+), 서비스 간 배포 주기가 다른 경우    |
| **장점**        | 서비스별 독립 배포, CI/CD 독립, 팀별 코드 소유권 명확 |
| **단점**        | 공유 타입 관리가 NPM 패키지 배포로 복잡해짐           |

### 선택 가이드

| 조건                            | 추천    |
| ------------------------------- | ------- |
| 1인 PO + AI 팀, 문서가 많음     | **A-1** |
| 개발자 2~5명, 문서 적음         | A-2     |
| 팀 10명+, 서비스 독립 배포 필요 | A-3     |

## B. 백엔드 아키텍처 패턴

### B-1. 모놀리스

| 항목            | 내용                                                                 |
| --------------- | -------------------------------------------------------------------- |
| **적합한 상황** | MVP, 사용자 1만명 이하, 빠른 개발이 중요                             |
| **장점**        | 개발·배포·디버깅이 단순. AI 에이전트가 전체 코드를 한 번에 파악 가능 |
| **단점**        | 규모 커지면 빌드 느려짐, 부분 배포 불가                              |
| **프레임워크**  | NestJS, Express, FastAPI, Django, Rails                              |

### B-2. 모듈러 모놀리스

| 항목            | 내용                                                               |
| --------------- | ------------------------------------------------------------------ |
| **적합한 상황** | 도메인이 3개 이상으로 나뉘고, 향후 마이크로서비스 전환 가능성      |
| **장점**        | 모놀리스의 단순함 + 모듈 간 경계 명확. 나중에 모듈을 떼어내기 쉬움 |
| **단점**        | 모듈 간 의존성 관리 필요                                           |
| **프레임워크**  | NestJS (모듈 시스템이 이 패턴에 최적화)                            |

### B-3. 마이크로서비스

| 항목            | 내용                                                 |
| --------------- | ---------------------------------------------------- |
| **적합한 상황** | 대규모 트래픽, 팀 10명+, 서비스별 독립 스케일링 필요 |
| **장점**        | 서비스별 독립 배포·스케일링                          |
| **단점**        | 운영 복잡도 급증. 1인 팀에 과도                      |

### B-4. 서버리스

| 항목            | 내용                                      |
| --------------- | ----------------------------------------- |
| **적합한 상황** | 트래픽 변동이 크고, 운영 인력 최소화 필요 |
| **장점**        | 인프라 관리 불필요, 사용량 기반 과금      |
| **단점**        | 콜드 스타트, 로컬 개발 환경 복잡          |

### 선택 가이드

| 조건                        | 추천    |
| --------------------------- | ------- |
| MVP, 빠른 개발, 1인 팀      | **B-1** |
| 도메인 3개+, 향후 확장 고려 | B-2     |
| 대규모 트래픽, 대형 팀      | B-3     |
| 운영 인력 0, 트래픽 변동 큼 | B-4     |

## D. 폴더 구조 템플릿

### D-1. A-1 + B-1 + C-1 (1인 PO + AI 팀 MVP에 최적)

> A-1 추천 이유: 기획 문서가 많은 환경에서 .docs/와 src/가 섞이면 관리가 어려움. 분리하면 "문서는 여기, 코드는 저기"가 명확해짐.

```
project/
├── COMPANY.md            ← 회사 정의
├── CLAUDE.md             ← AI 파트너 역할 + 프로젝트 정보
├── teams/                ← 팀 정의
├── agents/               ← 에이전트 정의
├── skills/               ← 스킬 정의
├── processes/            ← 업무 프로세스
├── rules/                ← 운영 규칙
├── references/           ← 참조 자료
├── hooks/                ← 자동화 안전장치
├── settings.json         ← 실행 설정
├── .docs/                ← 문서 산출물
│   ├── 000_inbox/
│   ├── 001_decision-making/
│   ├── 002_meeting-notes/
│   ├── 003_research/
│   ├── 004_planning/
│   │   ├── specs/
│   │   ├── screen/
│   │   ├── tech-spec/
│   │   ├── erd/
│   │   └── qa/
│   ├── 005_design/
│   └── 999_references/
├── src/                  ← 개발 산출물
│   ├── backend/
│   ├── frontend/
│   │   ├── mobile/
│   │   └── web/
│   ├── admin/
│   ├── shared/
│   │   ├── types/
│   │   ├── utils/
│   │   └── design-tokens/
│   ├── package.json
│   ├── pnpm-workspace.yaml
│   └── turbo.json
├── .gitignore
└── .prettierrc
```

### D-2. A-1 + B-2 + C-1 (도메인이 복잡한 서비스)

D-1과 동일하되, `src/backend/` 내부가 모듈러 구조:

```
src/
├── backend/
│   └── src/
│       ├── modules/        ← 도메인별 독립 모듈
│       │   ├── auth/
│       │   ├── payment/
│       │   └── notification/
│       ├── shared/         ← 모듈 간 공유
│       └── main.ts
├── frontend/
└── shared/
```

## E. 의사결정 체크리스트

### 1단계: 프로젝트 구조

- [ ] 문서와 코드를 분리할 필요가 있는가?
- [ ] 빌드 도구는? (pnpm workspace + Turborepo 기본 추천)

### 2단계: 백엔드 아키텍처

- [ ] MVP 단계인가?
- [ ] 도메인이 3개 이상으로 나뉘는가?
- [ ] 서비스별 독립 스케일링이 필요한가?

### 3단계: 프론트엔드 아키텍처

- [ ] 모바일 앱이 필요한가?
- [ ] 네이티브 성능이 핵심인가?

### 4단계: 폴더 구조 확정

- [ ] 위 선택 조합에 맞는 D 섹션 템플릿 선택
- [ ] CLAUDE.md에 최종 구조 기재
