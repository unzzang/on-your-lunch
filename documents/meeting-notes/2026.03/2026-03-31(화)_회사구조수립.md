# On-Your-Mark Company 구조 수립

## 논의 배경

PACE 프로젝트를 진행하며 개발 프로세스를 표준화했는데, 이를 확장하여 **서비스 프로덕트 + 콘텐츠를 개발·생산·운영하는 회사** 자체를 구축하기로 했다. Agent Companies 프로토콜과 Paperclip 오케스트레이션 개념을 참고하되, 우리 실전 경험을 기반으로 만들었다.

## 핵심 결정 사항

### 1. 회사명: On-Your-Mark

온라인 서비스 프로덕트와 콘텐츠(블로그, 유튜브)를 개발·생산·운영하는 가상 회사.

### 2. 조직 구성 — 8개 팀, 22명

| 팀        | 인원                                                                 |
| --------- | -------------------------------------------------------------------- |
| 리서치팀  | 6명 (조사설계, 시장, 경쟁, 고객, 실현가능성, 전략)                   |
| 디자인팀  | 2명 (김지수-서비스UI, 박소율-콘텐츠비주얼)                           |
| 개발팀    | 4명 (이명환-아키텍트, 이인수-백엔드, 잭도슨-프론트, 톰하디-관리자웹) |
| 콘텐츠팀  | 2명 (강서연-블로그, 류현우-영상)                                     |
| 마케팅팀  | 2명 (한지민-전략, 이수빈-SNS)                                        |
| 데이터팀  | 1명 (톰크루즈)                                                       |
| QA/검증팀 | 2명 (진도준-QA, 송현아-검증+팩트체크)                                |
| 운영팀    | 2명 (최민호-운영, 정예림-CS)                                         |

### 3. 운영 구조 — PO 직접 지시

- **기존:** PO → 위더 → 에이전트 (허브-스포크)
- **변경:** PO → 에이전트 직접 소통. 위더는 호출 시에만 개입.
- **이유:** PO가 에이전트와 대화하는 줄 알았는데 실제로는 위더가 가로채서 대신 일하고 있었음.

### 4. 위더의 역할 재정의

- 항상 활성 중재자 → **대기가 기본, 호출 시에만 개입**
- 개입하는 경우: PO 부재 대리, 복잡한 다팀 조율, 명시적 호출
- "함께하는 자" — PO의 파트너이지, 모든 업무의 관문이 아님

### 5. 폴더 구조 — 회사 vs 프로젝트 분리

| 회사 레벨     | 프로젝트 레벨 | 용도      |
| ------------- | ------------- | --------- |
| `.claude/`    | `.claude/`    | AI 인프라 |
| `.documents/` | `.docs/`      | 문서      |
| `.infra/`     | `src/`        | 코드      |
| `.efforts/`   | —             | 현황판    |
| `.kits/`      | —             | 템플릿    |

### 6. 에이전트 표준 형식 — Agent Companies 프로토콜 준수

- `reportsTo`: 파일 경로 기반 (`../팀장/AGENTS.md`)
- `skills`: 에이전트가 사용 가능한 스킬 slug
- 계층 구조: PO → AI파트너 → 팀장 → 팀원

### 7. Rules 구조 — 공통 + 팀별

- 공통 규칙 10개 (위임, 서술, Gate진입, 운영구조, 폴더, docs정의, efforts, documents, infra, kits)
- 팀별 규칙 8개 (리서치, 개발, 디자인, 콘텐츠, 마케팅, 데이터, QA, 운영)
- INDEX.md로 빠른 검색

### 8. Skills 구조 — 공식 규격

- 에이전트 스킬 15개 + 프로세스 스킬 7개 = 22개
- frontmatter 필수 (name + description with 트리거 키워드)
- 각 스킬에 참조 문서 테이블 포함
- INDEX.md로 빠른 검색

### 9. Processes — 서비스 7 Stage + 콘텐츠 + 마케팅

서비스: Stage 0(셋업) → 1(리서치) → 2(기획) → 3(디자인) → 4(개발) → 5(QA) → 6(출시)
콘텐츠: 키워드리서치 → 작성 → 검증 → 발행
마케팅: GTM → 채널운영 → 성과분석

### 10. .efforts/ — 프로젝트 현황판

칸반 보드 방식. 카드만 이동, 프로젝트 폴더는 이동하지 않음.
001_On → 002_Ongoing → 003_Simmering → 004_Sleeping → 005_Archive

### 11. .kits/ — 템플릿 + 첨부

- `templates/product/` — 서비스 프로젝트 스타터 킷
- `templates/documents/` — 문서 템플릿 (회의록, DR, 프로젝트 카드)
- `attachments/` — 외부 참고 파일 (PDF, 이미지 등)
