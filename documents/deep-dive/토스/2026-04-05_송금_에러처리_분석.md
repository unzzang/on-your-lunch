## 3. 7 Layers 심층 분석

### 3.1 플로우 해부 -- 송금 전체 흐름과 에러 분기점

#### 정상 송금 플로우 (출처: S4, S5)

```
[1] 홈 -> 송금 진입
    |
[2] 수신자 선택 (3가지 경로)
    +-- 계좌번호 직접 입력
    +-- 연락처 송금
    +-- 최근 송금 내역에서 선택
    |
[3] 계좌번호 입력 -> ML 기반 은행 자동 추천 (상위 3개, 정확도 99%+)
    |
[4] 금액 입력
    |
[5] 송금 정보 확인 (금액 / 받는 분 표시명 / 출금 계좌 / 입금 계좌)
    |
[6] 생체인증 또는 비밀번호 인증
    |
[7] 송금 처리 (로딩)
    |
[8] 송금 완료 화면
```

토스는 기존 "은행 선택 -> 계좌번호 입력" 순서를 역으로 변경하여 "계좌번호 입력 -> ML이 은행 3개 추천"으로 개선했다. (출처: S4) 사용자 인터뷰에서 62개 은행 목록에서 자기 은행을 찾지 못하고 오래 머무는 현상을 발견했고, "은행을 빨리 찾게 하려면?"이라는 질문을 "은행 찾기를 없앨 수 있을까?"로 전환한 결과이다.

#### 에러 분기점 맵

각 단계에서 발생 가능한 에러를 매핑한다.

```
[2] 수신자 선택 단계
    +-- 자기 자신에게 송금 시도 -> UX로 방지 (자신의 연락처 목록 미노출) [S1]
    +-- 차단된 연락처 -> 연락처 목록에서 미노출

[3] 계좌번호/은행 입력 단계
    +-- 존재하지 않는 계좌 -> 에러 메시지
    +-- 가상계좌/특수목적 계좌 -> 송금 불가 안내 [S5]
    +-- 사기 의심 계좌 -> 경고 다이얼로그 ("사기 의심 계좌예요") [S8, S13]

[4] 금액 입력 단계
    +-- 잔액 부족 -> 잔액 부족 안내
    +-- 1회 한도 초과 (200만원) -> 한도 안내 [S6]
    +-- 1일 한도 초과 (1,000만원) -> 한도 안내 [S6]
    +-- 은행 출금 한도 초과 -> 출금 한도 안내 [S5]

[6] 인증 단계
    +-- 생체인증 실패 -> 재시도 + 비밀번호 대체 안내 [S1]
    +-- 비밀번호 오류 -> 재입력 안내

[7] 송금 처리 단계
    +-- 네트워크 끊김 -> 네트워크 에러 화면
    +-- 은행 점검 시간 (23:30~00:30) -> 점검 안내 [S5]
    +-- 타임아웃 -> "처리 중" 상태
    +-- FDS 이상거래 탐지 -> 거래 차단 [S10, S14]
```

#### 핵심 설계 원칙

토스의 에러 처리 최우선 원칙은 **"최고의 에러는 발생하지 않는 것"**이다. (출처: S1)

구체적 사례: 연락처 송금에서 사용자가 자신의 연락처를 선택하면 에러가 발생했다. 토스는 에러 메시지를 개선하는 대신, 연락처 목록에서 자신의 연락처를 아예 노출하지 않는 방식으로 해결했다. 에러 메시지를 쓸 필요 자체를 없앴다.

### 3.2 마이크로카피 수집

#### 3.2.1 토스 에러 메시지 6가지 원칙 (출처: S1)

토스 UX Writer 김자유가 공개한 원칙이다.

| # | 원칙 | 설명 |
|---|------|------|
| 1 | **최고의 에러는 발생하지 않는 것** | 에러 메시지를 쓰기 전에 UX 개선으로 에러 자체를 제거할 수 있는지 먼저 검토 |
| 2 | **적절한 컴포넌트 쓰기** | 심각도에 따라 토스트/다이얼로그/전체 화면을 구분. 중요한 정보를 토스트로 표시하면 놓침 |
| 3 | **스스로 해결할 수 있는 방법 알려주기** | **가장 중요.** 사용자가 다음에 무엇을 해야 하는지 구체적 해결책 제시 |
| 4 | **사용자 입장에서 이해할 수 있는 언어** | 기술 용어/시스템 관점 표현 금지. 사용자가 이해할 수 있는 원인 제시 |
| 5 | **쉽게 해결할 수 있게 도와주기** | 탭 한 번으로 해결 화면에 도달하게 함. "고객센터 문의해주세요" 대신 바로 이동 버튼 |
| 6 | **부정적인 감정 최소화** | "~할 수 없어요" -> "~할 수 있어요" / "~해야 해요" -> "~할 수 있어요" |

#### 3.2.2 에러 메시지 3요소 공식 (출처: S1)

토스의 모든 에러 메시지는 반드시 3요소를 포함한다.

```
[상황 설명] + [이유] + [해결책]
```

#### 3.2.3 수집된 실제 에러 메시지 (Before -> After)

토스가 공식적으로 공개한 에러 메시지 개선 사례이다.

**사례 1: 생체인증 실패** (출처: S1)

| 구분 | 문구 |
|------|------|
| Before | "생체 인증에 실패했습니다" |
| After | "생체 인증에 실패했습니다. 앱을 재부팅하거나 재설치하면 해결됩니다" |
| 개선 | 원칙 3 -- 해결 방법 추가 |

**사례 2: 만료된 신분증** (출처: S1)

| 구분 | 문구 |
|------|------|
| Before | "알 수 없는 신분증입니다" |
| After | "만료된 신분증입니다" |
| 개선 | 원칙 4 -- 시스템은 "알 수 없는"으로 분류하지만, 사용자에게는 "만료된"이 이해 가능 |

**사례 3: 대출 재신청 불가** (출처: S1)

| 구분 | 문구 |
|------|------|
| Before | (토스트) 짧은 안내 후 3~5초 사라짐 |
| After | (다이얼로그) "심사가 시작돼서 앱 안에서 대출을 취소할 수 없습니다" + 충분한 설명 |
| 개선 | 원칙 2 -- 중요한 정보를 일시적 토스트가 아닌 확인형 다이얼로그로 변경 |

**사례 4: 하루 한 번 대출 신청 제한** (출처: S1)

| 구분 | 문구 |
|------|------|
| Before | "고객센터에 문의해주세요" |
| After | 고객센터 문의 화면으로 **바로 이동하는 버튼** 제공 |
| 개선 | 원칙 5 -- 사용자가 직접 찾아가야 하는 부담 제거 |

**사례 5: 사기 의심 계좌 경고** (출처: S8, S13)

| 항목 | 내용 |
|------|------|
| 경고 문구 | **"사기 의심 계좌예요"** |
| 노출 시점 | 송금 최종 단계 (확인 화면) |
| 컴포넌트 | 경고 다이얼로그 (전면 팝업) |
| 실제 사용자 반응 | 경고 확인 후 송금 중단. 온라인에서 "토스가 또 한 사람 구했다"라는 반응 (출처: S13) |

#### 3.2.4 송금 실패 유형별 안내 문구 (출처: S5)

토스 고객센터 FAQ에서 확인된 유형별 안내이다.

| 실패 유형 | 안내 문구 (원문) | 해결 안내 |
|----------|----------------|----------|
| **한도 초과** | "토스 송금 최대 한도(1회 200만원 / 1일 1,000만원) 이내 금액이어도 계좌 출금 한도를 넘기면 송금이 어려울 수 있어요" | 해당 은행에서 출금 한도 확인 |
| **특수목적 계좌** | "송금하려는 계좌가 가상계좌 혹은 청약, 적금 등 특수목적 계좌인 경우, 금액이나 시간에 따라 송금이 원활하지 않을 수 있어요" | 일반 입출금 계좌로 변경 |
| **은행 점검 시간** | "밤 늦은 시간이나 새벽 시간에 송금을 시도한 경우, 토스는 은행망을 이용하는 시스템이기 때문에 은행 점검 시간의 영향을 받습니다" | 점검 이후 재시도 |
| **잔액 부족 (자동이체)** | "잔액 부족으로 이체 실패 시, 즉시 알림을 보내드려요. 알림을 받으신 경우에는 계좌에 돈을 넣어도 자동이체가 되지 않으니 직접 송금해주세요" | 잔액 충전 후 직접 수동 송금 |

위 문구들의 공통 패턴: **"~할 수 있어요"** / **"~어려울 수 있어요"** 어미를 사용하여 강제가 아닌 안내 톤을 유지한다.

#### 3.2.5 토스 8가지 라이팅 원칙 (출처: S3)

에러 메시지를 포함한 모든 문구에 적용되는 상위 원칙이다.

| # | 원칙 | 핵심 |
|---|------|------|
| 1 | Predictable hint | 다음 화면에서 무슨 일이 일어날지 예측 가능한 문장 |
| 2 | Weed cutting | 의미 없는 단어 제거. 모바일에서 빠른 스캔 |
| 3 | Remove empty sentences | 같은 내용 반복하는 문장 제거 |
| 4 | Focus on key message | "지금 꼭 필요한 정보"만 추려서 전달 |
| 5 | Easy to speak | 전문 용어/문어체 금지. 말하듯이 자연스럽게 |
| 6 | Suggest over force | 사용자 선택권 존중. 공포/강요로 행동 유도 금지 |
| 7 | Universal words | 모두가 이해하는 표현. 유행어/밈 지양 |
| 8 | Find hidden emotion | 정보 뒤의 감정에 공감. 대출 완제 시 축하 메시지 |

#### 3.2.6 에러 메시지 시스템화 (출처: S2)

토스는 에러 메시지를 개별 작성이 아닌 시스템으로 관리한다.

| 구성 요소 | 설명 |
|----------|------|
| **에러 메시지 라이브러리** (개발자용) | 코드에서 에러 메시지 컴포넌트를 직접 호출. 디자이너-개발자 간 반복 소통 불필요 |
| **프레이머링** (디자이너용) | Framer 내에서 에러 메시지 템플릿 검색, 컴포넌트 단위 복사/붙여넣기 |
| **상황별 템플릿** | 은행 점검, 서비스 장애 등 자주 발생하는 에러에 변수(은행명, 시간)만 바꾸는 템플릿 |
| **핵심 가치** | "Navigating error" -- 에러 메시지의 역할은 다음 화면으로 넘어갈 수 있게 안내하는 것 |
| **크로스 체크** | 라이터들이 각 사일로를 방문해 메시지 개선 후, 비슷한 에러끼리 동일 문구로 규격화 |

### 3.3 상태별 대응

#### 상태 전이 다이어그램

```
[정상 입력] -> [확인 화면] -> [인증] -> [처리 중] -> [완료]
                                           |
                                      [실패/대기]
                                      +-- 즉시 실패 -> 에러 화면 + 해결책 버튼
                                      +-- 대기(Pending) -> "처리 중" 상태 유지 + 취소 가능
                                      +-- 지연 -> 알림 예약
```

#### 상태별 대응 상세

| 상태 | 화면 표현 | 사용자 액션 | 출처 |
|------|----------|-----------|------|
| **정상 (Normal)** | 송금 완료 화면 | 홈 복귀, 추가 송금 | 확인 불가 -- 완료 화면 세부 UI 미공개 |
| **로딩 (Loading)** | 처리 중 인디케이터 | 대기 | 확인 불가 -- 세부 UI 미공개 |
| **대기 (Pending)** | "대기중인 송금" 표시. '내 소비' 탭에서 확인 | 대기중인 건 터치 -> '송금 취소' 가능 | S5 |
| **실패 (Failed)** | 에러 다이얼로그 또는 전체 화면 | 원인별 해결 액션 버튼 | S1, S5 |
| **타임아웃** | "처리 중" 상태 유지 | 거래 내역에서 상태 확인 | S5 |
| **은행 점검** | 점검 시간 안내 | 예약 이체 또는 점검 후 재시도 | S5 |

#### 은행 점검 시간 처리 (출처: S5)

| 시간대 | 상태 | 토스 대응 |
|--------|------|----------|
| 매일 23:30~00:30 | 타행 정기 점검 | 송금/출금/카드결제 일시 불가 안내. 템플릿 시스템으로 은행명+시간 자동 삽입 |
| 은행별 비정기 점검 | 해당 은행 거래 불가 | 은행명 + 점검 시간 구체 안내 |

자동이체 설정 시 은행 점검 시간에 걸리면, 점검 종료 후 토스가 자동으로 송금 처리한다는 안내를 제공한다. (출처: S5)

### 3.5 인터랙션 패턴

#### 에러 컴포넌트 선택 기준 (출처: S1, S2)

토스는 에러의 심각도와 필요한 사용자 행동에 따라 컴포넌트를 구분한다.

## 4. 시사점 정리

모든 시사점은 [사실] -> [해석] -> [적용] 구조를 따른다.

### 시사점 1: 에러를 없애는 것이 최선의 에러 처리

**[사실]** 토스는 에러 메시지를 개선하기 전에 "이 에러가 꼭 있어야 하는가?"를 먼저 묻는다. 자기 자신에게 송금하는 에러를 에러 메시지가 아닌 연락처 목록 필터링으로 해결했다. (출처: S1)

**[해석]** 대부분의 팀은 "어떤 문구를 쓸까"에 집중한다. 토스는 한 단계 앞선 질문 -- "이 에러가 발생하는 상황 자체를 제거할 수 없는가?" -- 을 먼저 한다. UX 변경으로 에러를 제거하면 메시지를 아무리 잘 써도 달성할 수 없는 수준의 사용자 경험이 된다.

**[적용]** 온유어런치에서 발생 가능한 에러를 나열한 뒤 "UI/UX 변경으로 에러 자체를 제거할 수 있는 항목"을 먼저 식별한다. 예: 영업시간 외 주문 시도 -> 영업시간 외에는 주문 버튼을 비활성화하고 "내일 오전 11시에 열려요"로 대체.

### 시사점 2: 에러 메시지 = 상황 + 이유 + 해결책

**[사실]** 토스의 모든 에러 메시지는 3요소(상황, 이유, 해결책)를 반드시 포함한다. 그 중 "해결책"이 가장 중요하다. (출처: S1)

**[해석]** "오류가 발생했습니다"나 "다시 시도해주세요"는 범용 에러 메시지이다. 사용자에게 "무엇을 해야 하는지" 전혀 알려주지 않는다. 돈이나 시간이 관련된 에러는 사용자 불안을 극대화하므로, 구체적 해결 경로 제시가 필수다.

**[적용]** 온유어런치의 모든 에러 메시지에 3요소 공식을 적용한다. 예: "예약을 완료하지 못했어요(상황). 선택하신 시간이 방금 마감되었어요(이유). 다른 시간을 선택해보세요(해결책)."

### 시사점 3: 컴포넌트 = 심각도의 시각적 표현

**[사실]** 토스는 에러 심각도에 따라 토스트(가벼운) / 다이얼로그(중요) / 전체 화면(심각)을 구분한다. 중요한 정보를 토스트로, 가벼운 상황에 빨간색을 쓰는 것을 Anti-Pattern으로 정의했다. (출처: S1)

**[해석]** 컴포넌트 선택은 "어디에 표시하느냐"가 아니라 "이 에러가 얼마나 심각한지 사용자에게 어떻게 전달하느냐"의 문제이다. 토스트는 "괜찮아, 지나가도 돼"라는 시그널이고, 다이얼로그는 "잠깐, 이건 확인해야 해"라는 시그널이다.

**[적용]** 온유어런치의 에러 컴포넌트 기준을 아래와 같이 정의한다.
- **토스트**: 찜 추가/제거, 가벼운 성공 피드백
- **다이얼로그**: 예약 실패, 결제 오류, 식당 정보 변경
- **전체 화면**: 로그인 만료, 서버 다운, 인터넷 끊김

### 시사점 4: 해결 버튼은 탭 한 번으로

**[사실]** 토스는 "고객센터에 문의해주세요"라는 텍스트 대신 고객센터로 바로 이동하는 버튼을 제공한다. (출처: S1)

**[해석]** 에러 상황의 사용자는 이미 좌절감을 느끼고 있다. "직접 찾아가세요"는 추가 부담이다. 해결 경로를 최대한 짧게 만들어야 한다.

**[적용]** 온유어런치의 모든 에러 화면에서 해결 액션은 버튼으로 제공한다.
- 예약 실패 -> "다른 시간 보기" 버튼
- 식당 폐업 -> "비슷한 식당 보기" 버튼
- 네트워크 에러 -> "다시 시도" 버튼

### 시사점 5: 부정형을 긍정형으로

**[사실]** 토스는 "~할 수 없어요"를 "~할 수 있어요"로, "~해야 해요"를 "~할 수 있어요"로 전환한다. (출처: S1)

**[해석]** 부정형 문구는 "이 서비스가 내 문제를 해결하지 못한다"는 인상을 준다. 긍정형 전환은 "문제가 있지만 해결할 수 있다"는 메시지를 전달한다.

**[적용]** 온유어런치 에러 문구 작성 기준:
- "예약할 수 없습니다" -> "오후 2시 이후에 예약할 수 있어요"
- "검색 결과가 없습니다" -> "이 조건에 맞는 식당을 찾지 못했어요. 조건을 바꿔보세요"

### 시사점 6: 에러 메시지 시스템 관리

**[사실]** 토스는 에러 메시지를 라이브러리/템플릿 시스템으로 관리한다. 개발자가 코드에서 컴포넌트를 호출하고, 디자이너는 Framer에서 검색/복사한다. (출처: S2)

**[해석]** 서비스가 커지면 에러 메시지가 수백 개가 된다. 개별 관리하면 같은 유형의 에러에 다른 문구, 일관성 없는 톤이 된다. 시스템화하면 일관성 + 작성 시간 단축 + 크로스 체크 용이.

**[적용]** 온유어런치에서 에러 메시지를 상수 파일(constants/errorMessages.ts)로 중앙 관리한다. 카테고리별 템플릿을 정의하고 변수만 교체.

### 시사점 7: 사전 차단 + 사후 안전망 이중 구조

**[사실]** 토스는 사기 송금에 대해 사전(FDS + 사기 DB 대조)과 사후(착오송금 반환 + 지연이체 + 안심보상제) 두 겹 안전장치를 운영한다. 6개월간 15만건 차단, 160억원 피해 예방. (출처: S10, S11)

**[해석]** 사전 차단만으로는 모든 위험을 방지할 수 없고, 사후 안전망만으로는 피해가 이미 발생한다. 둘 다 있어야 사용자가 "안전하다"고 느낀다.

**[적용]** 온유어런치에서 식당 정보 오류(폐업, 가격 변경)에 대해 사전(식당 정보 최신성 배치) + 사후(오류 신고 -> 즉시 반영) 이중 구조를 설계한다.

### 시사점 8: 성숙기 제품의 문제는 비언어적

**[사실]** 토스는 은행 선택 UX를 개선할 때, 고객센터 피드백에는 불만이 없었지만 사용자 인터뷰에서 62개 은행 목록에서 오래 머무는 현상을 발견했다. (출처: S4)

**[해석]** 사용자는 "원래 이런 거 아닌가?"라고 생각하는 불편함을 직접 말하지 않는다. 데이터(체류 시간)와 관찰(인터뷰)로만 발견할 수 있다.

**[적용]** 온유어런치 출시 후, 에러 화면의 체류 시간과 이탈률을 추적한다. 에러 후 복귀율이 낮은 화면을 우선 개선 대상으로 식별한다.

## 5. 온유어런치 적용 제안

### 5.1 에러 메시지 작성 원칙 (토스 6원칙 기반)

| # | 원칙 | 온유어런치 적용 |
|---|------|---------------|
| 1 | 에러를 없앨 수 있으면 없앤다 | 영업시간 외 -> 버튼 비활성화. 품절 메뉴 -> 목록 제거 |
| 2 | 심각도에 맞는 컴포넌트 | 토스트/다이얼로그/전체 화면 3단계 기준 정의 |
| 3 | 해결 방법 반드시 제시 | 모든 에러에 {상황} + {이유} + {해결책} 공식 |
| 4 | 사용자 언어 | "서버 응답 없음" -> "지금 접속이 많아서 느려요" |
| 5 | 탭 한 번으로 해결 | 에러 화면에 해결 액션 버튼 필수 |
| 6 | 긍정형 어미 | "~없습니다" -> "~할 수 있어요" |

### 5.2 에러 컴포넌트 기준

| 심각도 | 컴포넌트 | 사용 사례 |
|--------|---------|----------|
| 하 | 토스트 (3초 자동 소멸) | 찜 추가/제거, 클립보드 복사 |
| 중 | 다이얼로그 (확인 버튼) | 예약 실패, 식당 정보 변경, 히스토리 삭제 |
| 상 | 전체 화면 에러 | 네트워크 끊김, 서버 다운, 인증 만료 |
| 인라인 | 필드 하단 텍스트 | 입력값 검증 실패 |

## 6. 확인 불가 항목 목록

실제 앱 조작 또는 내부 자료 없이는 검증할 수 없는 항목이다.

| 항목 | 확인 불가 이유 |
|------|-------------|
| 송금 완료 화면의 정확한 UI/애니메이션 | 공개 자료 없음 |
| 잔액 부족 시 에러 화면의 정확한 레이아웃 | FAQ 요약만 확인 가능 |
| 중복 송금 방지 다이얼로그 존재 여부 | 공개 자료 없음 |
| 인라인 에러(계좌번호 형식 오류)의 정확한 문구 | 공개 자료 없음 |
| 다크모드 에러 화면 색상 대비 | 실제 앱 테스트 필요 |
| VoiceOver 에러 메시지 읽기 순서 | 실제 앱 테스트 필요 |
| 에러 발생 시 햅틱 피드백 유무 | 실제 앱 테스트 필요 |
| 에러 analytics 이벤트 구조 | 내부 자료 필요 |
| 에러 메시지 라이브러리의 구체적 분류 체계 | 내부 자료 필요 |
| 사기 의심 경고 다이얼로그의 정확한 버튼 구성 | 스크린샷 미확보 |
