# 기획 프로세스 (Stage 2)

> 적용 범위: 서비스 정의부터 기술 설계까지
> 책임자: PO (의사결정), 위더 (기획 문서 작성)
> 선행 조건: Stage 1 Go 결정

---

## 1. 역할

서비스를 정의하고, 기능을 설계하고, 기술 아키텍처를 확정하여 개발 착수 가능한 상태를 만든다.

## 2. 프로세스

```
Gate 1. 서비스 정의 → Gate 2. 기능 설계 → Gate 3. 기술 설계
```

### Gate 1. 서비스 정의

| 활동 | 담당 | 산출물 |
|------|------|--------|
| 타겟 사용자 정의 | 위더 + PO | `.docs/004_planning/specs/` |
| 핵심 가치 + 차별점 | 위더 + PO | 상동 |
| BM 설계 | 위더 + PO | 상동 |
| 검증 | 송현아 | 피드백 |

### Gate 2. 기능 설계

| 활동 | 담당 | 산출물 |
|------|------|--------|
| 기능 맵 작성 | 위더 | `.docs/004_planning/specs/` |
| 기능 명세서 (기능별) | 위더 | 상동 |
| 화면 설계 문서 | 위더 | `.docs/004_planning/design/` |
| QA 관점 리뷰 | 진도준 | 피드백 |
| 검증 | 송현아 | 피드백 |

### Gate 3. 기술 설계

| 활동 | 담당 | 산출물 |
|------|------|--------|
| 아키텍처 설계 | 이명환 | `.docs/004_planning/specs/` |
| ERD 설계 | 이명환 + 이인수 | 상동 |
| API 설계 | 이명환 | 상동 |
| 데이터 설계 | 톰크루즈 | 상동 |
| 검증 | 송현아 | 피드백 |

아키텍처 설계와 데이터 설계는 병렬 수행 가능. `references/architecture-patterns.md` 참조.

## 3. 완료 기준

- [ ] 서비스 정의 확정 (Gate 1)
- [ ] 기능 명세서 + 화면 설계 완료 (Gate 2)
- [ ] 아키텍처 + ERD + API + 데이터 설계 완료 (Gate 3)
- [ ] 모든 Gate에서 검증 통과
- [ ] 주요 결정사항 DR 기록 완료
