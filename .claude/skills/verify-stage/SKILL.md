# Stage 검증 실행

Stage 완료 시 검증 절차를 자동 실행한다. rules/common/stage-verification.md의 규칙을 따른다.

## 작업 기준

### 실행 조건

- 사용자 또는 위더가 "Stage N 검증 요청" 시 실행
- 또는 담당 에이전트 작업 완료 후 자동 실행 (hook 연동)

### 검증 순서

1. **현재 Stage 확인** — 프로젝트 efforts 카드 또는 사용자 지시에서 현재 Stage 파악
2. **1차 전문 검증** — 해당 Stage에 전문 검증자가 있으면 투입
   - Stage 2: 이명환 (기술 스펙)
   - Stage 3: 김지수 (디자인 원칙)
   - Stage 4: 이명환 (코드 리뷰)
3. **2차 정합성 검증** — 1차 통과 후 투입
   - Stage 0~3, 6: 송현아 (산출물 정합성)
   - Stage 4~5: 진도준 (QA)
   - Stage 6: 송현아 + 진도준 (공동)
4. **검증 결과 보고** — PO에게 텔레그램 알림 + .docs/000_inbox/에 보고서 저장
5. **PO 승인 대기** — PO 승인 시 Stage 완료, 실패 시 수정 후 재검증

### 검증 실패 시

- 실패 사유를 담당 에이전트에게 전달
- 담당 에이전트가 수정 완료 후 재검증 요청
- 재검증은 실패한 검증 단계부터 재시작

## 보고 형식

```
[Stage N 검증 완료]

검증 대상: {Stage명}
1차 검증: {담당자} — {통과/실패} ({사유})
2차 검증: {담당자} — {통과/실패} ({사유})

발견 사항:
- (있으면 기재)

판정: 통과 / 조건부 통과 / 실패
PO 승인 필요
```

## 산출물

| 산출물 | 저장 위치 |
|--------|----------|
| 검증 결과 보고서 | `.docs/000_inbox/{날짜}({요일})_Stage{N}_검증결과.md` |

## 협업

- **송현아(검증전문가)** — 산출물 정합성, 출처, 모호 표현 검증
- **진도준(QA)** — 빌드/테스트, API 계약, 기능 검증
- **이명환(백엔드 아키텍트)** — 기술 스펙 타당성, 코드 리뷰
- **김지수(UI 디자이너)** — 디자인 원칙 준수, 토큰 사용 여부
- **PO** — 최종 승인
- **위더** — PO 부재 시 대리 승인
