# Phase 2. 화면 디자인

> 담당: 김지수. Phase 1의 디자인 시스템을 기반으로 모든 화면을 디자인한다.

---

## 선행 조건

- [ ] Phase 1 디자인 시스템 확정
- [ ] `docs/004_planning/screen/` 화면 설계 문서 존재
- [ ] `docs/004_planning/specs/` 기능 명세서 존재

## 절차

1. 기능 맵(`feature-map.md`)에서 MVP 핵심 플로우 순서를 확인한다
2. 화면 설계(`screen/`)를 읽고 각 화면의 구조/인터랙션을 파악한다
3. 기능 명세서(`specs/`)를 읽고 동작 규칙/에러 케이스를 파악한다
4. 핵심 플로우 순서로 화면을 디자인한다
5. 각 화면마다 **4가지 상태를 반드시 디자인**한다

### 4가지 상태 (누락 불가)

| 상태 | 설명 | 예시 |
|------|------|------|
| **정상 (Default)** | 데이터가 있는 정상 상태 | 추천 식당 3곳 표시 |
| **빈 상태 (Empty)** | 데이터 없음 | "조건에 맞는 식당이 없어요" |
| **로딩 (Loading)** | 데이터 로딩 중 | Skeleton UI |
| **에러 (Error)** | 네트워크/서버 에러 | "인터넷 연결을 확인해주세요" + 재시도 |

### 디자인 원칙 확인 (design-principles.md)

화면 디자인 시 아래를 체크한다:
- 한 화면 하나의 주 행동 (CTA 1개만 Primary)
- 정보 계층 3단계 이내
- 일관성 > 창의성
- 터치 영역 44×44px 이상
- 명암 대비 4.5:1 이상

## 디자인 파일 형식

| 형식 | 용도 | 저장 위치 |
|------|------|----------|
| **HTML/CSS** | 개발 참조용 (브라우저에서 inspect 가능) | `design/{화면명}.html` |
| **이미지 에셋** | 아이콘, 일러스트, 사진 | `design/assets/` |

HTML 형식을 사용하는 이유:
- 브라우저에서 inspect로 정확한 CSS 값 측정 가능
- QA Phase 3-D 디자인 검증 시 실제 구현과 나란히 비교 가능
- 반응형 확인 가능

## 산출물

- `design/{화면명}.html` — 화면별 디자인 파일 (4가지 상태 포함)
- `design/assets/` — 에셋 파일 (아이콘, 이미지)
