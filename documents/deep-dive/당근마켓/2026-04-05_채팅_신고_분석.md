## 3. Phase 2 — 심층 분석 (7 Layers)

### 3.1 Layer 1: 플로우 해부

당근에서 채팅과 관련된 신고는 크게 3가지 경로로 진입한다.

#### 경로 A: 채팅방 전체 신고

```
[채팅방] → [우측 상단 더보기(⋮)] → [신고하기] → [신고 사유 선택] → [상세 내용 입력] → [제출]
```

- **Step 1**: 문제 채팅방 진입
- **Step 2**: 우측 상단 세로 점 3개(⋮) 메뉴 탭
- **Step 3**: 메뉴에서 "신고하기" 선택
- **Step 4**: 신고 사유 카테고리 선택 (카테고리별로 별도 페이지 진입)
- **Step 5**: 상세 내용 텍스트 입력
- **Step 6**: 제출 (버튼 텍스트: "당근마켓 팀에게 보내기")

총 단계 수: 4~5단계 (진입점 2단계 + 사유 선택 1단계 + 내용 입력 1단계 + 제출 1단계)

> 출처: [S10], [S11], [S14]

#### 경로 B: 메시지별 신고 (2020년 12월 도입)

```
[채팅방] → [특정 메시지 롱프레스] → [신고하기] → [신고 사유 선택] → [제출]
```

- **Step 1**: 문제 메시지가 포함된 채팅방 진입
- **Step 2**: 해당 말풍선을 길게 누름 (롱프레스)
- **Step 3**: 컨텍스트 메뉴에서 "신고하기" 선택
- **Step 4**: 신고 사유 선택
- **Step 5**: 제출

총 단계 수: 3~4단계

> 출처: [S3], [S4]

#### 경로 C: 비매너 평가 (거래 완료 후)

```
[채팅방] → [우측 상단 더보기(⋮)] → [매너 평가하기] → [비매너 평가하기] → [항목 선택] → [완료]
```

- **Step 1**: 거래 상대방 채팅방 진입
- **Step 2**: 우측 상단 세로 점 3개(⋮) 탭
- **Step 3**: "매너 평가하기" 선택
- **Step 4**: "비매너 평가하기" 선택
- **Step 5**: 해당 비매너 항목 선택 (복수 선택 가능 여부 — 확인 불가)
- **Step 6**: 완료

총 단계 수: 4~5단계

> 출처: [S14], [S11]

#### 수집 정보 요약

| 경로 | 수집 정보 | 증거 첨부 |
|------|----------|----------|
| 채팅방 전체 신고 | 신고 사유 카테고리, 상세 설명 텍스트 | 확인 불가 |
| 메시지별 신고 | 특정 메시지 자체(자동 첨부 추정), 신고 사유 | 메시지 자동 캡처 추정 (확인 불가) |
| 비매너 평가 | 비매너 항목 선택 | 별도 증거 첨부 없음 |

### 3.2 Layer 2: 마이크로카피 수집

당근의 신고 관련 마이크로카피는 사용자의 심리적 부담을 줄이는 방향으로 설계되어 있다.

#### 확인된 마이크로카피

| 위치 | 원문 텍스트 | 톤 | 의도 |
|------|-----------|-----|------|
| 신고 제출 버튼 | "당근마켓 팀에게 보내기" | 친근, 협조적 | "신고하기"라는 직접적 표현 대신 "팀에게 보내기"로 부담감 경감 |
| 신고 사유 선택 후 안내 | "이 항목으로 신고하면 서로의 게시글이 보이지 않고, 서로 더 이상 채팅을 보낼 수 없어요" | 명확, 구체적 | 신고 결과를 사전 고지하여 신고자가 결과를 예측할 수 있게 함 |
| 차단 확인 | "네, 차단할게요" | 대화체 | "차단하기"가 아니라 대화형 확인으로 소통 감정 극대화 |
| 사기 이력 경고 (레드라벨) | "이 전화번호는 최근 3개월 동안 3회 이상 사이버범죄 신고시스템에 사기로 접수된 이력이 있는 전화번호입니다" | 경고, 사실적 | 구체적 기간과 횟수를 명시하여 위험 수준을 객관적으로 전달 |
| 채팅 중 실시간 경고 | 선입금 유도, 외부 채널 유도, 가입 정보와 다른 전화번호 공유 시 자동 경고 메시지 표시 | 경고 | 사기 패턴 실시간 감지 후 즉각 경고 |

> 출처: [S10] — UX Writing 분석, [S8] — 경찰청 연동 보도자료, [S9] — 소비자 보호 전략 보도

#### 마이크로카피 설계 원칙 분석

1. **부담감 경감**: "신고하기" → "당근마켓 팀에게 보내기"로 표현을 완화. 신고가 아닌 "도움 요청"의 프레이밍.
2. **결과 사전 고지**: 신고 전에 어떤 일이 발생하는지 구체적으로 명시 ("서로의 게시글이 보이지 않고...").
3. **대화체 사용**: "네, 차단할게요"처럼 사용자가 직접 말하는 느낌의 버튼 텍스트.
4. **객관적 경고**: 사기 이력 경고 시 "3개월", "3회 이상" 등 수치를 포함하여 모호함 배제.

### 3.3 Layer 3: 상태별 대응

#### 상태 전이 다이어그램

```
[정상 채팅 상태]
    │
    ├─ (사용자가 신고 진행) ──→ [신고 접수 중]
    │                              │
    │                              ├─ 운영팀 검토 ──→ [조치 완료]
    │                              │                    ├─ 경고
    │                              │                    ├─ 게시물 삭제
    │                              │                    ├─ 이용 정지 (3일~30일)
    │                              │                    └─ 영구 차단
    │                              │
    │                              └─ 허위 신고 판정 ──→ [기각]
    │
    ├─ (레드라벨 감지) ──→ [경고 표시 상태]
    │                        채팅방 + 프로필에 붉은색 경고 배너
    │
    ├─ (차단) ──→ [차단 상태]
    │              서로 게시글 비표시 + 채팅 불가
    │
    └─ (상대방 이용 제한) ──→ [제한 상태]
                               "이용이 제한되었습니다" 표시
```

#### 각 상태별 사용자 경험

| 상태 | 신고자 경험 | 피신고자 경험 |
|------|-----------|-------------|
| **정상 채팅** | 일반 채팅 기능 이용 가능 | 일반 채팅 기능 이용 가능 |
| **신고 접수 중** | 신고 후 즉각적 피드백 없음 (확인 불가 — 웹 검색 기반 분석의 한계) | 신고 사실을 인지하는지 여부 확인 불가 |
| **경고 조치** | 확인 불가 | 운영정책 위반 경고 수신 |
| **이용 정지** | 확인 불가 | 최소 3일 ~ 최대 30일 이용 정지 |
| **차단 상태** | 상대방 게시글 비표시, 채팅 불가 | 신고자 게시글 비표시, 채팅 불가 |
| **레드라벨 상태** | 채팅방에 붉은색 경고 배너 표시 | 프로필에 경고 표시, 거래 시도 시 상대방에게 경고 자동 노출 |

### 3.5 Layer 5: 인터랙션 패턴

#### 5.1 신고 진입점 접근성

| 진입점 | 접근 방법 | 탭 횟수 | 발견 용이성 |
|--------|---------|---------|-----------|
| 채팅방 더보기 메뉴 | 채팅방 → ⋮ → 신고하기 | 2탭 | 중간 (더보기 메뉴 안에 숨어 있음) |
| 메시지 롱프레스 | 메시지 길게 누르기 → 신고 | 1탭 + 롱프레스 | 높음 (문제 메시지에서 바로 접근) |
| 프로필 페이지 | 상대 프로필 → 신고 | 확인 불가 | 확인 불가 |
| 게시글 더보기 | 게시글 → ⋮ → 신고하기 | 2탭 | 중간 |

#### 5.2 사유 선택 UI 패턴

- **카테고리 분리형**: 체크박스가 아니라 카테고리별로 별도 페이지로 진입하는 구조 (출처: [S10])
- **1-depth 선택**: 대분류를 선택하면 해당 사유에 대한 설명 페이지가 표시됨
- **설명 + 결과 고지**: 사유를 선택하면 해당 사유에 대한 설명과 신고 후 발생할 일을 명시

#### 5.3 증거 첨부

| 증거 유형 | 지원 여부 | 비고 |
|----------|----------|------|
| 메시지 자동 캡처 | 추정 (메시지별 신고 시) | 롱프레스한 메시지가 자동 첨부될 것으로 추정 |
| 스크린샷 첨부 | 확인 불가 | 고객센터 문의 시에는 증거 자료 첨부 권장 |

## 4. Phase 3 — 시사점 정리

### 시사점 1: 심리적 부담 경감 마이크로카피

- **[사실]** 당근은 "신고하기" 대신 "당근마켓 팀에게 보내기"를 사용하고, "차단하기" 대신 "네, 차단할게요"를 사용한다 (출처: [S10]).
- **[해석]** "신고"라는 단어가 주는 법적·사회적 무거움을 의도적으로 제거한 설계. 중고거래 플랫폼에서 신고 행위의 진입장벽을 낮추는 효과. 이는 신고율을 높여 커뮤니티 자정 작용을 강화하려는 전략으로 판단된다.
- **[적용]** 커뮤니티 신뢰 기능 설계 시, 신고 CTA의 문구를 "운영팀에 알리기" 등으로 완화하면 신고 허들을 낮출 수 있다. 단, 신고 남용 리스크와의 균형 필요.

### 시사점 2: 결과 사전 고지 패턴

- **[사실]** 신고 사유 선택 시 "이 항목으로 신고하면 서로의 게시글이 보이지 않고, 서로 더 이상 채팅을 보낼 수 없어요"라는 결과를 미리 보여준다 (출처: [S10]).
- **[해석]** 신고 행위의 결과를 사전에 명시함으로써 (1) 신고자가 결과를 이해한 상태에서 의사결정하게 되고 (2) 실수로 인한 신고를 방지하며 (3) 신고 후 "왜 이렇게 됐지?"라는 혼란을 예방한다.
- **[적용]** 모든 제재성 액션(신고, 차단, 숨김 등)에 대해 실행 전 결과 프리뷰를 제공해야 한다. "이 행동을 하면 OO이 발생합니다" 패턴.

### 시사점 3: 카테고리 분리형 신고 UI

- **[사실]** 체크박스 나열이 아니라, 카테고리별로 별도 페이지로 진입하는 구조 (출처: [S10]).
- **[해석]** 카테고리 분리형은 (1) 각 사유에 대한 설명을 충분히 제공할 수 있고 (2) 신고자가 자신의 상황에 맞는 정확한 사유를 선택하도록 유도하며 (3) 운영팀이 사유별로 다른 처리 프로세스를 적용하기 용이하다.
- **[적용]** 신고 사유가 5개 이상이면 카테고리 분리형 UI 검토. 각 사유 선택 시 해당 사유의 정의와 예시를 함께 보여주는 패턴.

### 시사점 4: 다층 안전망 (자동 + 수동 + 외부 연동)

- **[사실]** 당근은 (1) 실시간 채팅 패턴 감지 (2) 경찰청 사기 이력 DB 실시간 연동 (3) 사용자 수동 신고 (4) 비매너 평가 누적 — 4개 계층의 안전망을 운영한다 (출처: [S8], [S9]).
- **[해석]** 단일 신고 채널만으로는 사기와 비매너를 막을 수 없다. 사전 방지(실시간 감지) + 사후 대응(신고 처리) + 외부 연동(경찰청)의 조합이 효과적. 특히 경찰청 연동은 플랫폼 내부 데이터만으로는 식별할 수 없는 전과자/상습범을 감지하는 데 효과적이다.
- **[적용]** 커뮤니티 신뢰 기능 MVP에서는 수동 신고 + 기본 패턴 감지로 시작하되, 규모 확장 시 외부 DB 연동을 로드맵에 포함해야 한다.

### 시사점 5: 신고 처리 투명성 부족 문제

- **[사실]** 당근은 개별 신고의 처리 결과를 신고자에게 직접 통보하는 시스템을 갖추고 있는지 확인되지 않는다. 고객센터 문의에 대해서는 12시간 내 응답 (출처: [S9]).
- **[해석]** 신고 결과 미통보는 "신고해봤자 소용없다"는 학습된 무력감을 유발할 수 있다. 반면 당근 측은 구체적 기준 공개가 악용될 수 있다는 우려를 표명 (출처: [S6]).
- **[적용]** 최소한 "신고가 접수되었습니다" → "검토 중입니다" → "조치가 완료되었습니다" 정도의 상태 전이 알림은 필요하다. 구체적 제재 내용(정지 N일 등)은 비공개로 유지하되, 처리 완료 여부는 알려주는 것이 사용자 신뢰 유지에 도움된다.

### 시사점 6: 허위 신고 방지의 구조적 한계

- **[사실]** 2022년 이전에는 채팅 없이도 아무나 비매너 신고 가능했으며, 이후 채팅 이력이 있어야만 평가 가능하도록 변경 (출처: [S6]). 가격 흥정 거절에 대한 보복 신고 사례 다수 보고됨 (출처: [S6]).
- **[해석]** 허위 신고 방지는 진입 조건 강화(채팅 이력 필수)와 사후 필터링(머신러닝)의 조합으로 접근하고 있으나, 주관적 판단 영역(비매너)에서는 근본적 한계가 존재한다. 당근이 기준을 비공개하는 것은 역이용 방지 목적이나, 동시에 피신고자의 구제 수단도 제한한다.
- **[적용]** 허위 신고 방지 설계 시: (1) 신고 가능 조건에 최소 상호작용 이력 요구 (2) 동일 사용자에 대한 단기간 반복 신고 제한 (3) 신고자의 신고 이력도 추적하여 악성 신고자 패턴 감지 (4) 이의 제기 채널 확보.

### 시사점 7: 안심보상 제도의 신뢰 구축 효과

- **[사실]** 2025년 3월 안심결제 이용 시 사기 피해 발생하면 최대 195만원까지 전액 보상. 무형 상품(티켓, 교환권)도 포함. 보상 횟수 무제한 (출처: [S12], [S13]).
- **[해석]** 신고 시스템만으로는 사후 구제가 한정적이다. 금전적 보상 장치를 결합하면 사용자의 거래 안전감이 크게 높아진다. "신고해도 돈은 못 돌려받는다"는 인식을 전환하는 효과.
- **[적용]** 거래형 커뮤니티 서비스라면, 신고 시스템 + 에스크로/보상 제도를 함께 설계해야 실질적 안전망이 된다.
