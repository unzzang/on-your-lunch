# 앱 서비스 분석 (App Service Analysis)

앱스토어 데이터와 실제 서비스 체험을 기반으로 특정 앱 또는 카테고리를 심층 분석한다.

## 실행 조건

- PO 또는 위더가 분석 대상 앱(또는 카테고리)을 지정
- 프로젝트 소속일 수도, 독립 분석일 수도 있음

## 5단계 프로세스

```
Phase 0: 분석 설계     — 위더 (대상, 범위, 분석 목적, 핵심 질문 정의)
Phase 1: 데이터 수집   — 앱분석전문가 (앱스토어 정보, 리뷰, 기사, 스크린샷)
Phase 2: 4영역 분석    — 앱분석전문가 (경쟁앱 / 카테고리 / UX기능 / 리뷰)
Phase 3: 검증          — 검증전문가 송현아 (출처 확인, 정합성, 논리 비약 체크)
Phase 4: 종합 보고서   — 앱분석전문가 (시사점 + 전략 제언)
```

---

## Phase 0: 분석 설계

위더가 PO와 함께 아래를 확정한다.

| 항목 | 내용 |
|------|------|
| **분석 대상** | 앱 이름 (복수 가능) |
| **분석 목적** | 왜 이 앱을 분석하는가 (경쟁 파악 / 벤치마크 / 시장 진입 검토 등) |
| **핵심 질문 (KRQ)** | 이 분석을 통해 답해야 할 질문 3~5개 |
| **분석 범위** | 4개 영역 중 전체 또는 선택 (경쟁앱 / 카테고리 / UX기능 / 리뷰) |
| **산출물 저장 위치** | 프로젝트 `.docs/003_research/app-analysis/` 또는 `.documents/app-analysis/` |

## Phase 1: 데이터 수집

앱분석전문가가 웹 검색/페치를 통해 원시 데이터를 수집한다.

### 수집 대상

| 소스 | 수집 항목 |
|------|----------|
| **App Store (iOS)** | 앱 정보, 평점, 리뷰, 스크린샷, 릴리스 노트, 카테고리 순위 |
| **Google Play (Android)** | 앱 정보, 평점, 리뷰, 다운로드 수, 업데이트 이력 |
| **기사/블로그** | 서비스 관련 보도, 인터뷰, 투자 정보, 사용자 후기 |
| **공식 웹사이트** | 기능 소개, 가격 정책, FAQ, 고객센터 |
| **SNS/커뮤니티** | 사용자 반응, 불만, 바이럴 콘텐츠 |

### 수집 규칙

- 모든 데이터에 **수집 날짜**와 **출처 URL** 기록
- 수집 불가 항목은 "데이터 없음 — 수집 방법: {제안}" 표기
- AI가 추정/생성한 수치 사용 절대 금지

## Phase 2: 4영역 분석

### 영역 1. 경쟁 앱 분석 (Store Intelligence)

| 분석 항목 | 세부 내용 | 분석 깊이 기준 |
|----------|----------|--------------|
| **기본 정보** | 앱 이름, 개발사, 출시일, 최종 업데이트일 | 업데이트 주기 계산 + 운영 활성도 판단 |
| **평점/리뷰** | 전체 평점, 1~5점 분포 비율, 총 리뷰 수 | 평점 추이 + 점수대별 주요 불만/호평 키워드 |
| **다운로드/순위** | 다운로드 수(Android), 카테고리 순위(iOS) | 시장 규모 추정 + 동일 카테고리 내 위치 |
| **가격 모델** | 무료/유료/인앱결제/구독, 가격대 | 수익 구조 분석 + 과금 전환 포인트 식별 |
| **업데이트 이력** | 최근 6개월 릴리스 노트 | 개발 투자 영역 추적 + 방향성 변화 해석 |
| **ASO 전략** | 제목, 부제, 키워드, 스크린샷 구성, 프리뷰 영상 | 타겟 포지셔닝 + 마케팅 메시지 분석 |
| **기술 정보** | 앱 크기, 지원 OS, 필요 권한 | 기술 복잡도 + 사용자 진입 장벽 |

### 영역 2. 카테고리 분석 (Market Landscape)

| 분석 항목 | 세부 내용 | 분석 깊이 기준 |
|----------|----------|--------------|
| **시장 구조** | 상위 20개 앱, 점유율 분포 | 과점/분산 판단 + 상위 3개 앱 특징 요약 |
| **트렌드** | 최근 6개월 신규 진입 앱, 순위 급상승 앱 | 상승 원인 분석 (기능? 마케팅? 시장 변화?) |
| **키워드 지형** | 카테고리 주요 검색 키워드, 경쟁 강도 | 블루오션 키워드 식별 + ASO 기회 |
| **평균 지표** | 카테고리 평균 평점/리뷰 수/업데이트 주기 | 기준선 설정 + 대상 앱의 상대적 위치 |
| **수익 모델 분포** | 무료/유료/구독/광고 비율 | 카테고리에서 검증된 수익 모델 식별 |
| **진입 장벽** | 네트워크 효과, 데이터 해자, 규제, 초기 투자 | 진입 가능성 판단 + 극복 전략 제언 |

### 영역 3. UX/기능 분석 (Product Teardown)

| 분석 항목 | 세부 내용 | 분석 깊이 기준 |
|----------|----------|--------------|
| **온보딩** | 첫 실행 흐름, 단계 수, 로그인 방식, 권한 요청 | 단계별 스크린샷 + 마찰 포인트 식별 |
| **핵심 기능** | 메인 기능 목록, 탭/메뉴 구성 | 기능별 접근 탭 수 + 우선순위 해석 |
| **IA(정보 구조)** | 탭 구성, 네비게이션 깊이, 전체 화면 수 | 사이트맵 시각화 + 복잡도 판단 |
| **핵심 플로우** | 가입→핵심기능→전환 경로 | 단계별 화면 + 이탈 위험 구간 표시 |
| **UI 패턴** | 컴포넌트, 인터랙션, 애니메이션 | 트렌드 준수 여부 + 차별화 요소 |
| **디자인 시스템** | 컬러, 타이포, 아이콘, 간격 | 브랜드 톤앤매너 요약 + 일관성 평가 |
| **빈 상태/에러** | 데이터 없음, 네트워크 끊김, 에러 화면 | 완성도 등급 (상/중/하) + 스크린샷 |
| **개인화** | 추천 여부, 개인화 깊이, 설정 항목 | 기술 투자 수준 판단 |
| **알림 전략** | 푸시 빈도, 유형, 시간대 | 리텐션 전략 해석 + 피로도 판단 |

### 영역 4. 리뷰 분석 (Voice of Customer)

| 분석 항목 | 세부 내용 | 분석 깊이 기준 |
|----------|----------|--------------|
| **감성 분류** | 긍정/부정/중립 비율 | 시간별 추이 + 감성 전환 시점 원인 분석 |
| **주제 분류** | 언급 주제별 빈도 (기능, UI, 속도, 가격 등) | 상위 5개 주제 + 주제별 대표 리뷰 인용 |
| **불만 패턴 TOP 10** | 반복 불만 순위, 빈도, 심각도 | 불만별 "우리에게 의미하는 바" 해석 |
| **요구사항 추출** | "~했으면", "~추가해주세요" 패턴 | 요구사항 목록화 + 실현 가능성/우선순위 |
| **버전별 반응** | 업데이트 전후 감성 변화 | 어떤 변경이 긍정/부정 반응을 유발했는지 |
| **충성 사용자** | 5점 리뷰 공통 키워드 | 경쟁사 핵심 강점 = 우리 필수 요건 |
| **이탈 신호** | "삭제", "탈퇴", "안 씀" 패턴 | 치명적 이탈 트리거 식별 |

## Phase 3: 검증

검증전문가(송현아)가 아래를 체크한다.

| 체크 항목 | 기준 |
|----------|------|
| **출처 확인** | 모든 수치에 출처(URL, 수집 날짜)가 있는가 |
| **가짜 데이터** | AI가 생성/추정한 수치가 없는가 |
| **논리 비약** | "A이므로 B이다"의 근거가 충분한가 |
| **누락** | 분석 설계의 KRQ에 모두 답했는가 |
| **모호 표현** | "적절한", "충분한" 등 모호 표현이 없는가 |

검증 실패 시: 앱분석전문가에게 수정 사항 전달 → 수정 → 재검증. PO에게 보고하지 않는다.

## Phase 4: 종합 보고서

검증 통과 후, 앱분석전문가가 종합 보고서를 작성한다.

### 종합 보고서 구조

```markdown
# {앱이름} 앱 서비스 분석 — 종합 보고서

## 분석 개요
- 분석 대상 / 분석 일자 / 분석 목적 / KRQ

## 핵심 발견 (Executive Summary)
- 3~5줄로 가장 중요한 발견 요약

## KRQ별 답변
- KRQ 1: {질문} → {답변 + 근거}
- KRQ 2: ...

## SWOT 요약
| | 긍정 | 부정 |
|---|---|---|
| 내부 | Strengths | Weaknesses |
| 외부 | Opportunities | Threats |

## 전략 시사점
- 반드시 해야 할 것 (Must-Have): 경쟁사가 잘하는 것 중 필수
- 차별화 기회 (Differentiator): 경쟁사가 못하는 것 중 우리가 할 수 있는 것
- 하지 말아야 할 것 (Avoid): 경쟁사가 실패한 것, 리뷰에서 불만인 것

## 데이터 출처 목록
- 수집한 모든 소스의 URL + 수집 날짜
```

## 실행 규칙

- 위더는 Phase 0만 수행하고, Phase 1~4는 앱분석전문가에게 위임한다
- Phase 3 검증은 반드시 검증전문가(송현아)가 수행한다
- 각 Phase는 순차 진행한다 (병렬 금지 — 수집 완료 후 분석, 분석 완료 후 검증)
- 가짜 데이터 절대 금지 — 출처 없는 수치는 "데이터 없음" 표기
