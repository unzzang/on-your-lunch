## 2. 핵심 발견사항 (Key Findings)

### 시장 영역 (출처: 시장분석 보고서)

1. **시장 규모가 크고 지속 성장 중이다.** 공동주택 관리비 시장은 2024년 기준 30.6조 원이며, 2019년 대비 5년간 53% 성장했다(연평균 +8.9%). 의무관리단지가 20,805개, 1,188만 세대에 달한다. (출처: 한국아파트신문)

2. **위탁관리 비율이 매우 높다.** 단지 기준 84%, 세대 기준 88.4%가 위탁관리를 채택하고 있다. 1위 업체 우리관리가 801개 단지를 관리하며, 상위 4개 업체가 각 384~801개 단지를 관리하는 집중 구조다. (출처: 아파트관리신문, 매거진한경)

3. **디지털화가 빠르게 진행 중이다.** 아파트아이 전자투표 참여율 99% 이상(2024년 약 6만 건), 전자고지서 3개월 만에 12만 명 신청, 삼성물산 홈닉 2.0이 2,990세대에서 56,000세대로 확대되는 등 아파트 관리의 모바일화가 시장 표준이 되어가고 있다. (출처: 한국아파트신문)

4. **관리업체의 IT 투자 여력이 제한적이다.** 672개 등록 업체 중 상위 20개를 제외하면 대부분 영세하며, 위탁관리 수수료 전국 평균이 세대당 월 600원으로 추가 소프트웨어 비용에 대한 저항이 높을 수 있다. (출처: 아파트관리신문, 다음 카페)

### 경쟁 영역 (출처: 경쟁분석 보고서)

1. **입주민 앱 영역은 레드오션이다.** 아파트아이(30,000여 단지, 1,200만 세대)와 아파트너(4,000여 단지, 320만 입주민)가 공지, 관리비, 투표, 민원, 방문차량 등 입주민 기능을 이미 광범위하게 제공하고 있다. (출처: 아파트아이/아파트너 공식 홈페이지)

2. **"다단지 통합 관리 대시보드"는 명확한 점유자가 없다.** 4개 직접 경쟁사(아파트아이, 아파트너, 직방 우리집, 아파트리) 모두에서 위탁관리업체가 복수 단지를 하나의 대시보드로 통합 관리하는 기능을 확인하지 못했다. 관리 ERP(이노가드, 제온스)도 회계/정산 중심으로 입주민 소통 기능이 약하다. (출처: 앱스토어 분석, 관리 ERP 홈페이지 조사)

3. **앱스토어 리뷰에서 반복되는 5가지 불만 패턴이 발견되었다.** (1) 단지 미등록 시 기능 불가, (2) 전자투표 참여율 문제, (3) UX 불편(메뉴 순서, 복수 날짜 예약 불가), (4) 관리사무소의 앱 미활용, (5) 주차 민원 압도적(33%). (출처: App Store 리뷰, 소비자경제, 아파트관리신문)

4. **해외에서는 다단지 통합 관리가 검증된 비즈니스 모델이다.** AppFolio(미국)는 복수 부동산 포트폴리오를 하나의 대시보드에서 통합 관리하는 기능을 핵심으로 제공하며, 이것이 전문 관리 회사의 핵심 니즈임을 증명한다. (출처: AppFolio 공식 사이트)

### 고객 영역 (출처: 고객분석 보고서)

1. **관리업체(B)의 5가지 핵심 페인포인트가 확인되었다.** 민원 폭주(하루 20건+), 공지 개별 발송 비효율, 관리비 정산 투명성 압박, 투표 행정 부담, 영세 업체 시스템 부재. 다단지 관리 시 이 모든 비효율이 단지 수만큼 배가된다. (출처: 한국아파트신문, 코리안포털뉴스)

2. **입주민(C)의 5가지 핵심 페인포인트가 확인되었다.** 공지 확인 누락, 관리비 투명성 불만(51%가 '비싸다' 응답), 민원 처리 불투명, 서면투표 불편, 앱의 단지 종속성. (출처: 한국아파트신문, 이투데이)

3. **전자투표 도입 효과가 입증되었다.** 투표율 평균 16% 상승, 비용 77% 절감, 인력 투입시간 64% 절감. 한 아파트에서 투표율이 19.1%에서 54.8%까지 상승한 사례가 있다. (출처: 아파트관리신문)

4. **"다단지 통합 대시보드"에 대한 관리업체의 직접적 수요 표현은 미발견이다.** 비효율은 다수 확인되었으나, 관리업체 스스로가 "통합 대시보드로 해결하고 싶다"고 호소하는 사례는 발견되지 않았다. 이는 리서치 방법의 한계(2차 데이터 기반)에 기인할 수 있다. (출처: 고객분석 H-1 검증)

### 실현가능성 영역 (출처: 실현가능성 보고서)

1. **기술적으로 구현 가능하다.** React Native + Next.js + Node.js + PostgreSQL은 표준 기술 스택이며, 멀티테넌시는 SaaS 표준 패턴이다. K-apt 공공 API를 통해 단지 기본정보를 확보할 수 있다. (출처: 공공데이터포털, AWS/Azure 아키텍처 가이드)

2. **전자투표가 법적으로 인정된다.** 공동주택관리법 제22조에 의해 전자투표의 법적 효력이 인정되며, 휴대전화 본인인증(건당 50~100원)으로 본인확인 요건을 충족할 수 있다. 다만 무료 공공 서비스(아파트 e투표)가 이미 존재한다. (출처: 공동주택관리법, LH e투표)

3. **MVP 비용은 시나리오에 따라 48만~8,360만 원이다.** 1인 개발 시 6개월 48만~160만 원, 외주 시 4,348만~8,360만 원. 월 운영비 약 8~10만 원(Supabase + Vercel + FCM). (출처: 위시켓, 크몽, Supabase/Firebase 가격표)

4. **핵심 병목은 관리비 정산 시스템 연동이다.** 관리업체마다 다른 ERP/정산 시스템을 사용하며, 표준 연동 규격이 없다. MVP에서는 수동 입력/CSV 업로드로 우회해야 한다. (출처: 실현가능성 5-1절)

## 3. 가설 검증 종합

### H-1: 위탁관리업체는 다단지 관리에서 시스템 분산으로 인한 비효율을 겪고 있다

| 항목 | 내용 |
|------|------|
| 판정 | **부분 확인** |
| 가설 | 3개 이상 단지를 관리하는 위탁관리업체는 단지별로 서로 다른 시스템을 사용하며, 공지 발송, 민원 처리, 관리비 정산 등에서 반복 업무가 발생한다 |
| 검증 결과 | 비효율 사례 다수 확인됨. 하루 20건+ 민원, 단지별 개별 채널(네이버 카페/카카오톡/게시판/앱)로 공지 개별 발송, 엑셀 기반 수기 정산 등. 그러나 관리업체가 "통합 대시보드"라는 해결책을 직접 요구하는 사례는 발견되지 않음 |
| 근거 보고서 | 고객분석(H-1 부분 확인), 경쟁분석(White Space 존재 확인), 시장분석(672개 업체 중 영세 다수) |
| Go/No-Go | **조건부 Go.** 비효율은 존재하나, "통합 대시보드"에 대한 직접 수요는 파일럿을 통해 검증해야 한다 |

### H-2: 기존 아파트 관리 앱은 단일 단지 중심으로 설계되어 있어 관리업체 관점의 통합 기능이 부재하다

| 항목 | 내용 |
|------|------|
| 판정 | **조건부 확인** |
| 가설 | 아파트아이, 우리아파트 등 기존 앱은 입주민-단일 관리사무소 간 소통에 초점이며, 관리업체가 복수 단지를 하나의 인터페이스로 관리하는 기능이 없거나 미흡하다 |
| 검증 결과 | 4개 직접 경쟁사 모두에서 다단지 통합 관리 기능 "확인하지 못함". 관리자 앱(아파트너 V2, 아파트리 관리사무소)이 존재하나 설치 수(10,000+)가 개별 관리사무소 단위 사용을 시사. 관리 ERP(이노가드, 제온스)는 회계 중심으로 입주민 소통 기능 약함. 다만, 기존 앱의 관리자 기능을 직접 사용해보지 못한 한계가 있어 "이미 제공 중일 가능성"을 완전히 배제할 수 없음 |
| 근거 보고서 | 경쟁분석(조건부 Go), 시장분석(아파트아이 1,200만 세대 선점 지적) |
| Go/No-Go | **조건부 Go.** 아파트아이/아파트너에 직접 문의하여 관리업체 대상 다단지 통합 기능 존재 여부를 확인해야 한다 |

### H-3: 수도권 500세대 이상 아파트의 위탁관리 비율은 B2B2C 모델을 성립시킬 만큼 높다

| 항목 | 내용 |
|------|------|
| 판정 | **확인 (Go)** |
| 가설 | 수도권 500세대 이상 아파트 중 위탁관리업체에 관리를 맡기는 비율이 과반 이상이며, 소수의 관리업체가 다수의 단지를 관리하는 구조가 존재한다 |
| 검증 결과 | 위탁관리 비율 단지 기준 84%, 세대 기준 88.4%로 Go 기준(50%)을 대폭 상회. 1위 우리관리 801개 단지, 상위 4개 업체 각 384~801개 단지로 Go 기준(10개 단지)을 대폭 상회. B2B2C 모델의 구조적 전제 조건이 성립한다 |
| 근거 보고서 | 시장분석(H-3 Go 충족) |
| Go/No-Go | **Go.** 조건 없이 충족 |

### H-4: 입주민은 관리 앱을 통한 공지 확인, 투표 참여, 민원 접수에 실질적인 수요가 있다

| 항목 | 내용 |
|------|------|
| 판정 | **확인 (Go)** |
| 가설 | 수도권 중대형 아파트 입주민은 관리사무소 소통을 모바일로 처리하길 원하며, 기존 방식에 불편을 느끼고 있다 |
| 검증 결과 | 전자투표 도입 시 투표율 16% 상승, 참여자 99%가 전자투표 선택. 민원 1위 주차(33%), 관리비 투명성 불만 51%. 아파트너 275만 세대, 아파트아이 1,200만 세대로 시장 자체가 성장 중. 다만 앱 활성 사용률(MAU) 데이터는 공개되지 않아, "설치만 하고 안 쓴다"는 시나리오를 배제할 수 없음. 전자투표의 95.45%가 알림톡/문자, 앱은 3.91%에 불과한 점은 별도 앱보다 문자/카카오톡 채널이 채택률에 유리함을 시사 |
| 근거 보고서 | 고객분석(H-4 확인), 시장분석(전자투표 99%, 전자고지서 12만 명) |
| Go/No-Go | **Go.** 단, 앱 자체의 활성 사용률 리스크를 설치 유도 전략으로 대응해야 한다 |

## 4. SWOT 분석

### 강점 (Strengths)

| # | 강점 | 근거 |
|---|------|------|
| S1 | 다단지 통합 관리라는 명확한 차별점 후보 | 기존 경쟁사 4개 모두에서 다단지 통합 기능 미확인 (경쟁분석) |
| S2 | B2B2C 구조의 영업 효율성 | 상위 관리업체 1곳 계약으로 수백 개 단지에 동시 진입 가능 (시장분석: 1위 업체 801개 단지) |
| S3 | 관리업체 내부 도구로의 진입 시 입주자대표회의 승인 불필요 | 관리업체 자체 사용 소프트웨어는 독자 결정 가능 (시장분석 4-1절) |
| S4 | 표준 기술 스택으로 MVP 구현 가능 | React Native + Next.js + Node.js + PostgreSQL, 멀티테넌시 SaaS 표준 패턴 (실현가능성 보고서) |

### 약점 (Weaknesses)

| # | 약점 | 근거 |
|---|------|------|
| W1 | 다단지 통합 관리에 대한 관리업체의 직접적 수요 미검증 | H-1이 "부분 확인"으로, 비효율은 존재하나 직접 수요 표현은 미발견 (고객분석) |
| W2 | 관리비 정산 시스템 연동 불가 (MVP 단계) | 관리업체마다 다른 ERP/정산 시스템 사용, 표준 규격 부재 (실현가능성 5-1절) |
| W3 | B2B 영업 인력/역량 부재 | 관리업체에 소프트웨어를 판매하려면 영업 인력과 ROI 입증이 필요 (실현가능성 6-4절) |

## 5. 핵심 리스크

### 리스크 1: 다단지 통합 관리에 대한 수요가 관리업체의 결제 전환으로 이어지지 않을 가능성

- **심각도:** 높음
- **근거:** H-1이 "부분 확인"이다. 비효율은 존재하나, 관리업체가 이를 해결하기 위해 유료 소프트웨어를 도입할 의향이 있는지는 미검증. 위탁관리 수수료 세대당 600원이라는 낮은 마진 구조에서 추가 IT 비용에 대한 저항이 높을 수 있다. (출처: 고객분석 H-1, 시장분석 4-2절)
- **대응:** MVP 전 관리업체 1~2곳 대상 무료 파일럿을 진행하여 실제 사용 반응과 결제 전환 의향을 검증한다. 파일럿 없이 전체 개발에 착수하지 않는다.

### 리스크 2: 선발주자(아파트아이/아파트너)의 기능 추격

- **심각도:** 높음
- **근거:** 아파트아이가 1,200만 세대, 아파트너가 320만 입주민을 이미 보유하고 있다. 이들이 "다단지 통합 관리" 기능을 추가하면, 위리브의 핵심 차별점이 사라진다. 아파트너 관리자 V2 앱이 이미 존재하며, 기능 범위를 확인하지 못한 상태다. (출처: 경쟁분석 7장, 검증결과 H-2)
- **대응:** (1) 아파트아이/아파트너 관리자 기능을 직접 확인(문의 또는 테스트). (2) "다단지 통합" 단일 기능이 아닌, 민원 SLA 워크플로우 + 단지 간 벤치마킹 등 관리업체 관점의 깊은 기능으로 차별화. (3) 속도 우선 — 선발주자가 기능 추가하기 전에 관리업체와 계약을 확보한다.

### 리스크 3: 입주민 앱 활성 사용률이 낮을 가능성

- **심각도:** 중간
- **근거:** 4개 보고서 모두 앱 MAU 데이터 부재를 공통 리스크로 지적. 전자투표의 95.45%가 알림톡/문자이고 앱은 3.91%에 불과. 별도 앱 설치에 대한 피로감, 고령 세대의 디지털 격차가 장벽. (출처: 고객분석 4-2절, 아파트관리신문)
- **대응:** (1) 입주민 접점의 핵심 기능(공지 알림, 투표)을 푸시 알림 + 카카오톡 알림톡으로도 제공하여 앱 미설치자도 참여 가능하게 설계. (2) 방문차량 등록, 관리비 조회 등 "일상적으로 반복 사용하는 기능"으로 앱 설치 유인을 만든다.

### 리스크 4: 알스퀘어 홈닷ERP와의 경쟁

- **심각도:** 중간
- **근거:** 2024년 12월 알스퀘어(연 매출 2,000억 원)가 공동주택 ERP 시장에 진출. 자본력과 영업망에서 위리브 대비 압도적 우위. (출처: 실현가능성 4-4-1절, 한국일보)
- **대응:** 알스퀘어 홈닷ERP는 관리비 정산/회계/세무에 집중. 위리브는 "입주민 소통 + 다단지 통합 관리"에 집중하여 포지셔닝을 분리한다. 장기적으로 홈닷ERP와 연동하는 보완재 전략도 검토한다.

### 리스크 5: 관리비 정산 시스템 연동의 기술적 난이도

- **심각도:** 중간 (단, MVP에서는 회피 가능)
- **근거:** 관리업체마다 상이한 정산 시스템 사용, 표준 연동 규격 부재. K-apt 관리비 데이터는 월별 파일 데이터로 실시간 API가 아님. (출처: 실현가능성 5-1절)
- **대응:** MVP에서 관리비 "정산"은 제외하고 "조회"만 제공. 관리업체가 CSV/Excel 업로드 또는 직접 입력하는 방식을 채택. 이후 표준 연동 모듈을 단계적으로 개발한다.

## 6. 전략적 시사점

### 6-1. 타겟 전략: B2B2C 구조에서 B(관리업체)를 먼저 공략한다

**1차 타겟: 수도권에서 5~30개 단지를 관리하는 중견 위탁관리업체**

상위 4개 대형 업체(384~801개 단지)는 자체 시스템을 보유할 가능성이 높고, 의사결정 구조가 복잡하여 초기 영업 대상으로 부적합하다. 반대로 5개 미만 소형 업체는 통합 관리 니즈 자체가 약하다.

5~30개 단지를 관리하는 중견 업체가 타겟인 이유:
- 단지 수가 충분하여 "통합 관리"의 가치를 체감할 수 있다
- 자체 IT 시스템이 없어(엑셀/수기 기반) 도입 가치가 명확하다
- 의사결정자(대표 또는 본사 운영팀장)에게 직접 접근 가능한 규모다

**2차 타겟: 1차 타겟 관리업체 산하의 입주민**

관리업체(B)가 도입한 후, 산하 단지의 입주민(C)에게 앱을 배포한다. B2B2C 구조이므로 입주민 개별 마케팅 비용이 0원에 수렴한다. 관리사무소가 "공지 확인을 위해 앱을 설치해주세요"라고 안내하는 구조.

### 6-2. 차별화 전략: 다단지 통합 + 관리 업무 체계화

**단순히 "다단지 통합 대시보드"만으로는 차별점이 약하다.** 기존 앱이 기능을 추가하면 쉽게 따라올 수 있다. 차별화를 심화하려면 "관리업체의 업무 프로세스 자체를 체계화하는 도구"로 포지셔닝해야 한다.

차별점 3가지:

1. **다단지 통합 대시보드:** 복수 단지의 공지/민원/투표/관리비를 하나의 화면에서 관리. 공통 공지 일괄 발송. (경쟁분석 White Space 1)

2. **민원 워크플로우 시스템:** 접수→담당자 배정→처리→완료→피드백의 체계적 워크플로우. SLA(접수 후 X시간 내 1차 응답) 설정과 모니터링. 관리사무소에 민원 처리 매뉴얼이 없는 문제를 시스템으로 해결. (경쟁분석 White Space 2)

3. **단지 간 데이터 벤치마킹:** 관리업체 산하 단지 간 민원 발생 빈도, 투표 참여율, 관리비 수준 비교. 데이터 기반 의사결정 도구. (경쟁분석 White Space 3)

### 6-3. MVP 범위 권고

**포함 (MVP 필수):**

| 기능 | 대상 | 근거 |
|------|------|------|
| 공지사항 작성/발송 + 푸시 알림 | B + C | 관리업체 페인포인트 A-2, 입주민 페인포인트 B-1 |
| 전자투표 (생성/참여/집계 + 본인인증) | B + C | 법적으로 인정, 투표율 16% 상승 효과 입증 (고객분석) |
| 민원 접수/처리 (워크플로우) | B + C | 민원 폭주(하루 20건+)와 추적 불가 문제 해결 (고객분석) |

**제외 (MVP 이후):**

| 기능 | 이유 |
|------|------|

| 방문차량 등록/주차 관리 | 아파트너가 이미 월 350만 회 이용. 차별점이 아님 |
| 커뮤니티 게시판 | 네이버 카페/카카오톡이 대안으로 존재. MVP에서 우선순위 낮음 |
| 택배 관리 | 핵심 가치(다단지 통합)와 무관한 부가 기능 |
| 관리비 카드결제 | 아파트아이가 이미 제공(1년간 3,500억 원 납부). 결제 시스템 구축 비용 대비 차별점 약함 |

### 조건: 아래 3가지를 충족해야 Go

**조건 1. 관리업체 파일럿을 먼저 진행한다 (H-1 수요 검증)**

MVP 전체 개발에 착수하기 전에, 수도권 중견 위탁관리업체(5~30개 단지) 1~2곳을 대상으로 무료 파일럿을 진행한다. 파일럿 범위는 관리자 웹 대시보드(다단지 통합 조회 + 공지 일괄 발송)의 프로토타입이다.

검증해야 할 것:
- 관리업체가 다단지 통합 대시보드를 실제로 사용하는가?
- 사용 시 업무 시간이 실측 가능한 수준으로 줄어드는가?
- 유료 전환 의향이 있는가? (세대당 200~500원 또는 단지당 20만~50만 원)

파일럿에서 위 3가지 중 하나라도 No가 나오면, MVP 방향을 재조정하거나 피벗을 검토한다.

**조건 2. 기존 앱의 관리자 기능을 직접 확인한다 (H-2 검증 보완)**

아파트아이와 아파트너에 관리업체 담당자로 문의하여, 다단지 통합 관리 기능의 존재 여부를 직접 확인한다. 이미 제공 중이라면 위리브의 차별점을 "민원 워크플로우 + 단지 간 벤치마킹"으로 재설정한다.

**조건 3. "맛집 앱"의 교훈을 적용한다 — 앱 전에 수동 테스트**

온유어런치 사전조사에서 도출된 "앱 개발 전 수동 테스트" 원칙을 적용한다. 관리업체 1곳에 구글 시트 + 카카오톡 알림톡 조합으로 "공지 통합 발송 + 민원 추적"을 수동으로 제공해보고, 실제 반응을 확인한 후 MVP 개발에 착수한다.

### 다음 단계 (Conditional Go 충족 시)

1. **즉시:** 아파트아이/아파트너 관리자 기능 직접 확인 (1주)
2. **1~2주:** 수도권 중견 위탁관리업체 2~3곳 컨택, 파일럿 제안
3. **3~6주:** 구글 시트 + 알림톡 기반 수동 파일럿 진행 (관리업체 1곳, 3~5개 단지)
4. **파일럿 결과 확인 후:** MVP 개발 착수 (공지 + 투표 + 민원 + 다단지 대시보드)

### 피벗 방향 (조건 미충족 시)

- **시나리오 A: 다단지 통합 수요가 약한 경우** — "단일 단지 관리 소통 앱"으로 피벗. 민원 워크플로우에 집중하여 개별 관리사무소에 영업. 다만 아파트아이/아파트너와 직접 경쟁하므로 차별점 약함.
- **시나리오 B: 기존 앱이 이미 다단지 기능을 제공하는 경우** — "관리업체 ERP 보완재"로 피벗. 관리비 정산/회계 특화 도구(알스퀘어 홈닷ERP와 경쟁)로 방향 전환. 다만 자본력 열세.
- **시나리오 C: 관리업체의 지불 의향이 없는 경우** — "입주민 직접 앱"으로 피벗. B2B2C 대신 B2C 모델. 다만 입주민 앱 영역은 레드오션이므로, 특정 기능(투표 전문, 커뮤니티 전문)에 니치 집중 필요.
