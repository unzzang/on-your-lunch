# 위리브 실현 가능성 분석 보고서

## 1. 핵심 기술 실현성

### 1-1. 기술 스택 선택지

위리브는 입주민용 모바일 앱(iOS/Android)과 관리자용 웹 대시보드 두 가지 클라이언트를 필요로 한다. 각 영역의 기술 스택 선택지를 비교한다.

#### 모바일 앱 (입주민용)

| 항목 | React Native | Flutter |
|------|-------------|---------|
| 언어 | JavaScript/TypeScript | Dart |
| iOS/Android 동시 대응 | 가능 | 가능 |
| 생태계 | npm 패키지 풍부. 토스, 리디, 인프런 등 국내 대형 서비스 사용 중 | Google 지원. 위젯 기반 UI. 국내 채택 증가 추세 |
| 웹과 코드 공유 | React 기반이므로 Next.js(관리자 웹)와 TypeScript/로직 공유 가능 | Flutter Web 존재하나 웹 대시보드용으로는 미성숙 |
| 푸시 알림 | FCM(Firebase Cloud Messaging) 연동 표준화 | FCM 연동 표준화 |
| MVP 적합성 | 높음. 웹 개발자가 바로 진입 가능 | 높음. 단, Dart 학습 필요 |

출처: [토스 React Native 기술](https://toss.tech/article/react-native-2024), [인프런 모바일 앱 개발 파트](https://tech.inflab.com/20240425-mobile-part/), [리디 React Native 전환 후기](https://ridicorp.com/story/react-native-1year-review/)

**권장:** React Native(Expo). 관리자 웹(Next.js)과 TypeScript를 공유할 수 있어 개발 인력 효율이 높다. React 개발자가 React Native 작업을 빠르게 수행할 수 있다는 점이 소규모 팀에 유리하다.

#### 관리자 웹 대시보드

| 항목 | Next.js | Remix | Vue/Nuxt |
|------|---------|-------|----------|
| React Native와 코드 공유 | TypeScript + 로직 공유 가능 | TypeScript + 로직 공유 가능 | 불가 (프레임워크 상이) |
| 대시보드 적합성 | 높음. SSR/SSG 지원 | 높음 | 높음 |
| 생태계 | Vercel 호스팅 최적화, 풍부한 UI 라이브러리 | 상대적으로 작은 생태계 | 한국 커뮤니티 활발 |

**권장:** Next.js. React Native와 TypeScript/shared-types를 공유하여 API 계약을 통일할 수 있다.

#### 백엔드

| 항목 | Node.js (Express/Fastify) | Go | Python (FastAPI) |
|------|--------------------------|----|--------------------|

**권장:** Node.js (Fastify) + TypeScript. 프론트엔드(React Native, Next.js)와 타입을 공유하여 API 계약 불일치를 방지한다.

### 1-2. 푸시 알림 시스템

공지사항 알림은 위리브의 핵심 기능이다. 입주민이 공지를 놓치지 않도록 푸시 알림을 발송해야 한다.

| 항목 | FCM (Firebase Cloud Messaging) | APNs (Apple Push Notification Service) |
|------|-------------------------------|---------------------------------------|
| 비용 | 무료 (텍스트 메시지 기준) | 무료 |
| 플랫폼 | Android + iOS (FCM이 APNs를 래핑) | iOS 전용 |
| 대량 발송 | 지원. 토픽(topic) 기반 구독으로 단지별 그룹 발송 가능 | FCM 경유 시 동일 |
| 이미지 첨부 | 가능. 단, 1MB 이미지를 100만 기기에 발송 시 약 $150 발생 | 가능 |

출처: [Firebase Pricing](https://firebase.google.com/pricing), [FCM 푸시 파헤치기](https://zuminternet.github.io/FCM-PUSH/), [FCM vs APNs 비교](https://appmaster.io/ko/blog/apns-vs-fcm-ios-android-pusi-alrim)

**위리브 적용 시 비용 추정:**

500세대 아파트 1개 단지 기준, 세대당 평균 1.5명이 앱을 설치한다고 가정하면 약 750명. 하루 공지 1~2건 발송 시 월간 약 22,500~45,000건의 텍스트 푸시 발송. FCM 무료 쿼터 내에서 충분히 소화 가능하다. 10개 단지(7,500명)로 확장해도 월 45만 건 수준으로 무료 범위 내다.

**권장:** FCM 단일 사용. React Native + FCM 연동은 표준화된 패턴이며, 추가 비용 없이 MVP 운영 가능하다.

### 1-3. 투표 시스템

#### 기술적 요건

| 요건 | 구현 방법 | 난이도 |
|------|----------|--------|
| 1인 1표 보장 | 세대 인증 + 투표 참여 기록 DB 관리 | 낮음 |
| 익명 투표 | 투표 내용과 투표자 정보를 분리 저장 (투표 테이블에 voter_id 미포함, 별도 참여 기록 테이블에서 참여 여부만 관리) | 중간 |
| 기명 투표 | 투표 내용에 voter_id 포함 | 낮음 |
| 실시간 집계 | 투표 시 집계 카운터 증가 (Redis 또는 DB atomic increment) | 낮음 |
| 투표 기간 관리 | 시작/종료 시간 DB 관리 + 스케줄러(cron) | 낮음 |
| 이중 투표 방지 | DB unique constraint (user_id + vote_id) | 낮음 |
| 투표 결과 무결성 | 투표 완료 후 결과 해시 저장, 수정 불가 처리 | 중간 |

**결론:** 투표 시스템은 표준적인 웹 개발 기술로 구현 가능하다. 블록체인 수준의 무결성이 아닌, DB 레벨의 무결성 보장이면 MVP에 충분하다.

#### 법적 효력과의 관계

공동주택관리법 제22조에 따라 전자투표는 법적으로 인정된다(후술 2-1 참조). 단, 본인확인 절차를 충족해야 한다. 위리브가 자체 전자투표 시스템을 구축할 경우, 법적 본인확인 요건(휴대전화 인증, 공인전자서명 등)을 반드시 구현해야 한다.

### 1-4. 다단지 통합 관리 — 멀티테넌시 아키텍처

위리브의 핵심 차별점인 "다단지 통합 관리"는 멀티테넌시(Multi-Tenancy) 아키텍처로 구현한다.

#### 멀티테넌시 모델 비교

| 모델 | 설명 | 장점 | 단점 | 위리브 적합성 |
|------|------|------|------|-------------|
| 단일 DB, 공유 스키마 | 모든 단지가 하나의 DB, tenant_id 컬럼으로 구분 | 운영 비용 최저, 구현 단순 | 데이터 격리 약함, 대규모 시 성능 이슈 | MVP에 적합 |
| 단일 DB, 스키마 분리 | 단지별 별도 스키마 (PostgreSQL schema) | 데이터 격리 중간, 마이그레이션 관리 용이 | 스키마 수 증가 시 관리 복잡 | 성장기에 적합 |
| 단지별 별도 DB | 단지마다 독립 DB 인스턴스 | 완전한 데이터 격리 | 운영 비용 높음, 커넥션 관리 복잡 | 과도함 |

출처: [AWS Multi-Tenant SaaS Architecture](https://aws.amazon.com/blogs/architecture/lets-architect-building-multi-tenant-saas-systems/), [Azure SaaS Multitenant Solution Architecture](https://learn.microsoft.com/en-us/azure/architecture/guide/saas-multitenant-solution-architecture/), [Multi-Tenant Architecture Best Practices](https://relevant.software/blog/multi-tenant-architecture/)

**권장:** MVP 단계에서는 "단일 DB, 공유 스키마 + tenant_id" 방식을 채택한다. PostgreSQL의 Row-Level Security(RLS)를 활용하면 tenant_id 기반 데이터 격리를 DB 레벨에서 강제할 수 있다. Supabase가 PostgreSQL RLS를 기본 지원하므로 인프라 선택과도 부합한다.

**기술적 난이도:** 중간. 멀티테넌시 자체는 SaaS 표준 패턴이나, 권한 체계(관리업체→단지→관리사무소 직원→입주민)가 4계층으로 복잡하다. 권한 설계에 주의가 필요하다.

#### 연동 가능 범위

| 데이터 | API로 확보 가능 여부 | 비고 |
|--------|:---:|------|
| 단지 기본정보 (주소, 세대수, 동수) | O | 공동주택 기본정보 API |
| 관리방식 (위탁/자치) | O | 공동주택 기본정보 API |
| 관리비 내역 (항목별) | 부분적 | 파일 데이터로 제공. 실시간 API 아님 |

**결론:** K-apt API를 통해 단지 기본정보와 관리방식 데이터를 확보할 수 있다. 이를 위리브 초기 셋업(단지 등록)에 활용 가능하다. 단, 개별 세대의 관리비 상세 데이터는 공개 API로 제공되지 않으므로, 관리업체가 직접 데이터를 입력하거나 관리업체의 기존 정산 시스템과 연동해야 한다.

### 1-6. 입주민 인증 방식

| 방식 | 구현 난이도 | 정확도 | MVP 적합성 |
|------|------------|--------|------------|
| 관리사무소 승인 방식 | 낮음 | 높음 | 적합. 입주민이 앱에서 세대 정보 입력 → 관리사무소 대시보드에서 승인/거부 |
| 초대 코드 방식 | 낮음 | 중간 | 적합. 관리사무소가 세대별 초대 코드 발급 → 입주민이 코드 입력 |
| 휴대전화 본인인증 | 중간 | 높음 | 전자투표 시 법적 본인확인 요건 충족에 필요. PASS 인증 등 외부 서비스 연동 필요 (건당 약 50~100원) |
| 우편 인증 | 높음 | 높음 | 비용과 시간 소요. MVP에 부적합 |

**권장:** MVP에서는 "관리사무소 승인 방식"을 기본으로 채택한다. 전자투표 기능 사용 시에만 휴대전화 본인인증을 추가 요구한다.

## 2. 규제/법적 요건

### 2-1. 공동주택관리법 — 전자투표

#### 법적 근거

공동주택관리법 제22조(전자적 방법을 통한 의사결정)에 따라, 입주자 등은 동별 대표자나 입주자대표회의의 임원을 선출하는 등 공동주택의 관리와 관련하여 전자적 방법을 통해 의사를 결정할 수 있다.

출처: [공동주택관리법 제22조 (LBOX)](https://lbox.kr/v2/statute/%EA%B3%B5%EB%8F%99%EC%A3%BC%ED%83%9D%EA%B4%80%EB%A6%AC%EB%B2%95/%EB%B3%B8%EB%AC%B8%20%3E%20%EC%A0%9C3%EC%9E%A5%20%3E%20%EC%A0%9C2%EC%A0%88%20%3E%20%EC%A0%9C22%EC%A1%B0)

#### 본인확인 요건 (시행령 제22조 제1항)

전자투표 시 아래 3가지 본인확인 방법 중 하나를 충족해야 한다:

| 방법 | 내용 | 위리브 구현 가능성 |
|------|------|-----------------|
| 1. 휴대전화 본인인증 | 정보통신망법 제23조의3에 따른 본인확인기관 제공 인증 (PASS, 카카오 인증 등) | 가능. 외부 본인확인 서비스 연동 필요 (건당 50~100원) |
| 2. 공인전자서명 | 전자서명법에 따른 공인전자서명/공인인증서 | 가능. 단, 사용자 편의성 낮음 |
| 3. 관리규약에서 정한 방법 | 관리규약에서 별도 본인확인 절차를 정한 경우 해당 방법 | 관리규약 개정이 선행되어야 함 |

출처: [공동주택 전자투표 모바일 뉴스](https://www.newsro.kr/article243/83729/), [아파트너스 전자투표 관련 민원](https://aptners.com/bbs/board.php?bo_table=minwon&wr_id=3139)

#### 기존 전자투표 서비스 현황

중앙공동주택관리지원센터(국토교통부 산하)가 운영하는 '아파트 e투표'가 2018년 12월부터 무료로 서비스되고 있다. 이 서비스는 PC와 스마트폰을 통해 투표할 수 있으며, 모든 내용을 암호화 처리하여 비밀투표 원칙을 준수한다.

출처: [아파트 e투표 (LH)](https://e-vote.lh.or.kr/), [중앙공동주택관리지원센터 e투표](https://myapt.molit.go.kr/evote/webEvoteMain.do)

**위리브에 대한 시사점:**

- 전자투표는 법적으로 인정된다. 이는 Go 요인이다.
- 단, 무료 공공 서비스(아파트 e투표)가 이미 존재한다. 위리브의 투표 기능이 차별화되려면, 단순 투표 기능이 아니라 "관리 대시보드 내 통합 투표 관리"(투표 생성→알림 발송→실시간 현황→결과 보고서 자동 생성)를 제공해야 한다.
- 본인확인 요건 충족을 위해 휴대전화 인증 서비스 연동이 필수다.

### 2-2. 관리비 공개 의무

공동주택관리법 시행령 제23조에 따라, 관리주체는 관리비 항목별 산출내역을 다음 달 말일까지 해당 아파트 단지의 인터넷 홈페이지와 K-apt(공동주택관리정보시스템)에 공개해야 한다. 2024년 기준으로 의무 공개 대상이 기존 300세대 이상에서 **50세대 이상**으로 확대되었다.

**위리브에 대한 시사점:**

- 관리비 공개가 법적 의무이므로, 위리브에서 관리비 내역을 입주민에게 보여주는 기능은 규제와 부합한다.

### 4-2. 위리브 수익 모델 선택지

| 수익 모델 | 과금 방식 | 장점 | 단점 | 적합성 |
|-----------|----------|------|------|:------:|
| **A. 관리업체 월 구독 (단지당)** | 관리업체가 관리하는 단지 수 x 월 구독료 | 매출이 관리 단지 수에 비례하여 확장. 관리업체 의사결정 단순 | 소수 단지 관리업체의 지불 의향 낮을 수 있음 | 높음 |
| **B. 세대당 과금** | 관리하는 총 세대 수 x 세대당 월 요금 | 대규모 단지에서 매출 극대화. 세대수와 서버 비용이 비례하므로 합리적 | 세대당 단가가 낮아야 관리비 항목으로 전가 시 저항 적음 | 높음 |
| **C. 프리미엄 기능 과금** | 기본(공지+투표) 무료, 프리미엄(민원관리+관리비+통계) 유료 | 초기 도입 장벽 낮음 | 무료 사용자만 남을 리스크 | 중간 |
| **D. 입주민 관리비에 포함** | 관리비 항목 중 "일반관리비" 또는 별도 항목으로 앱 사용료 포함 | 입주민 개별 결제 불필요 | 입주자대표회의 의결 필요. 관리비 증가에 대한 입주민 반발 가능 | 중간 |

#### 관리비 항목 포함 가능성

공동주택관리법 시행령 제23조에서 정하는 관리비 항목에 "소프트웨어 사용료"가 직접 명시되어 있지 않다. 다만, "일반관리비"에 포함시키거나 입주자대표회의 의결을 통해 별도 항목으로 설정할 가능성이 있다. 법적으로 명확하지 않으므로, 관리비 포함 방식보다는 관리업체(B2B)에 직접 과금하는 모델이 법적 리스크가 낮다.

### 4-3. 수익 추정 (단지당/세대당 과금 시나리오)

아래는 수도권 위탁관리업체 대상 B2B SaaS 모델을 가정한 추정이다. 실제 과금 금액은 시장 검증 후 결정해야 하며, 아래 수치는 해외 유사 서비스의 가격대를 참고한 산출 범위이다.

#### 시나리오 A: 단지당 과금

| 항목 | 보수적 | 적극적 |
|------|--------|--------|
| 단지당 월 구독료 | 20만 원 | 50만 원 |
| 초기 고객 (단지 수) | 10개 단지 | 30개 단지 |
| 월 매출 | 200만 원 | 1,500만 원 |
| 연 매출 | 2,400만 원 | 1억 8,000만 원 |

산출 근거: 해외 서비스 Buildium의 Growth 플랜이 $183/월(약 25만 원)이며, 한국 시장의 가격 민감도를 감안하여 20만~50만 원 범위로 설정. 알스퀘어 홈닷ERP도 세대수 기반 구독형이나 구체적 가격은 비공개.

#### 시나리오 B: 세대당 과금

| 항목 | 보수적 | 적극적 |
|------|--------|--------|
| 세대당 월 요금 | 200원 | 500원 |
| 초기 고객 (총 세대 수) | 5,000세대 (500세대 x 10단지) | 15,000세대 (500세대 x 30단지) |
| 월 매출 | 100만 원 | 750만 원 |
| 연 매출 | 1,200만 원 | 9,000만 원 |

산출 근거: 해외 property management SaaS의 유닛당 과금이 $1~$5/유닛/월이나, 한국 시장에서는 200~500원/세대/월이 현실적 수준으로 추정. 세대당 200원은 500세대 단지에서 월 10만 원에 해당하며, 관리비 대비 미미한 수준이다.

### 4-4. 수익 모델의 구조적 고려사항

1. **알스퀘어 홈닷ERP의 진입:** 2024년 12월 알스퀘어(부동산 데이터 기반 프롭테크, 연 매출 2,000억 원 규모)가 공동주택 ERP 사업에 진출했다. 자본력과 영업망을 갖춘 대형 플레이어의 진입은 위리브에 경쟁 위협이 된다. 단, 알스퀘어 홈닷ERP는 관리비 정산/회계/세무에 초점을 맞추고 있어, 위리브의 "입주민 소통 + 다단지 통합 관리"와 포지셔닝이 다르다.

## 6. 종합 판단

### 6-1. "만들 수 있는가?" — 기술적으로 가능하다

- React Native(모바일) + Next.js(관리자 웹) + Node.js(백엔드) + PostgreSQL(DB)은 표준적인 기술 스택이다.
- 멀티테넌시는 SaaS 표준 패턴이며, Supabase의 PostgreSQL RLS로 MVP 수준에서 구현 가능하다.
- 푸시 알림(FCM)과 투표 시스템은 표준적인 웹 기술로 구현 가능하다.
- K-apt 공공 API를 통해 단지 기본정보를 확보할 수 있다.
- **핵심 병목:** 관리비 정산 시스템과의 자동 연동. 관리업체마다 시스템이 달라 표준 연동이 어렵다. MVP에서는 수동 입력/CSV 업로드로 우회한다.

### 6-2. "규제를 통과할 수 있는가?" — 통과 가능하다

- 전자투표는 공동주택관리법 제22조에 의해 법적으로 인정된다. 본인확인 요건(휴대전화 인증)을 충족하면 법적 효력이 있다.
- 관리비 공개는 법적 의무이므로, 위리브의 관리비 조회 기능은 규제와 부합한다.
- 개인정보보호법 준수를 위해 동의 절차와 접근 통제를 구현해야 하나, 이는 표준적인 개발 공수다.
- **주의사항:** 관리비 체납자 정보의 무단 공개는 법적 처벌 대상이므로, 접근 권한 통제를 반드시 구현해야 한다.

### 6-3. "비용은 감당 가능한가?" — 시나리오에 따라 다르다

| 시나리오 | 6개월 총비용 | 판단 |
|----------|-------------|------|
| 1인 창업자가 직접 개발 | 48만~160만 원 | 감당 가능. 단, 6~10개월의 개발 기간과 모바일+웹+백엔드 역량 필요 |
| 자체 개발 (2인 팀) | 4,448만~7,060만 원 | 시드 투자 또는 상당한 자기 자본 필요 |
| 외주 개발 | 4,348만~8,360만 원 | 외주 관리 리스크 포함. 투자 유치 필요 |

온유어런치(맛집 앱) 대비 복잡도가 높아 비용이 약 1.5~2배이나, B2B SaaS 모델이므로 단위 고객당 매출(ARPU)이 높아 투자 대비 수익 회수 가능성은 더 높다.

### 6-4. "수익을 낼 수 있는가?" — 구조적으로 가능하나, 영업이 관건이다

- B2B SaaS 구독 모델(단지당 또는 세대당 과금)은 해외 유사 서비스에서 검증된 모델이다.
- 한국의 알스퀘어 홈닷ERP도 세대수 기반 구독형 모델을 채택했다.
- 단, 한국 아파트 관리 시장은 가격 민감도가 높고, 기존 무료 서비스(아파트아이 등)에 익숙한 관리업체를 유료 전환시키려면 명확한 ROI(투자 대비 효과)를 제시해야 한다.
- **경고:** 알스퀘어(연 매출 2,000억 원)가 동일 시장에 진출했다. 자본력과 영업망에서 열세인 상황에서, "다단지 통합 관리"라는 차별점이 실제 관리업체의 결제 전환으로 이어지는지 빠르게 검증해야 한다.

### 6-5. 권장 사항

1. **MVP는 핵심 기능 3가지(공지+투표+민원)에 집중한다.** 관리비 정산 연동은 MVP 이후 과제로 남긴다. 관리비 "조회"만 제공하고, "정산"은 기존 시스템에 맡긴다.

2. **전자투표 본인인증은 법적 요건이므로 MVP에 반드시 포함한다.** 휴대전화 본인인증 서비스 연동 비용(건당 50~100원)은 투표 빈도가 낮으므로(월 1~2회) 부담 가능하다.

3. **관리업체 1곳과 파일럿을 먼저 진행한다.** 5~10개 단지를 관리하는 중소 위탁관리업체 1곳과 무료 파일럿을 진행하여, "다단지 통합 관리"에 대한 실제 반응을 검증한다. 파일럿 없이 전체 개발에 착수하면 시장 검증 없이 비용이 소진된다.

4. **K-apt 공공 API를 활용하여 단지 셋업을 자동화한다.** 관리업체가 단지를 추가할 때, K-apt API에서 단지 기본정보(세대수, 동수, 주소)를 자동으로 불러오면 초기 셋업 비용을 줄일 수 있다.

5. **알스퀘어 홈닷ERP와의 포지셔닝을 명확히 한다.** 홈닷ERP가 "관리비 정산/회계/세무"에 집중하는 동안, 위리브는 "입주민 소통 + 다단지 통합 관리"에 집중한다. 장기적으로는 홈닷ERP와 연동하는 보완재 전략도 고려할 수 있다.
