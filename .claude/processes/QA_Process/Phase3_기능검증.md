# Phase 3. 기능 검증

> 실제로 동작하는지 검증한다. API + 브라우저 UI + 디자인 대조.

---

## 3-A. API 기능 검증

모든 API 엔드포인트에 curl로 요청하여 응답을 확인한다.

- [ ] 전체 엔드포인트 정상 응답 (200/201)
- [ ] 에러 케이스 응답 확인 (400, 401, 404, 409)

---

## 3-B. UI 인터랙션 검증 (브라우저 필수)

**브라우저에서 실제 사용자 동작을 수행**하여 기능이 동작하는지 확인한다. API가 정상이어도 프론트에서 연결이 안 되면 **기능 미구현**으로 판정한다.

**담당:** 위더 (브라우저 검증) + 개발팀 (발견 이슈 수정)

서브 에이전트는 Bash 권한 제한으로 브라우저(browse) 실행이 불가하므로, **위더가 브라우저 검증을 수행**하고, 발견된 이슈의 코드 수정은 개발팀에 위임한다.

**검증 방법:**

1. dev 서버 실행 (`pnpm dev`)
2. browse 스킬로 모바일 뷰포트(393x852) 접속
3. 각 기능을 직접 조작하고 결과 확인
4. 동작 전/후 스크린샷 촬영
5. 이슈 발견 시 → 이슈 목록 정리 → 개발팀에 수정 위임 → 수정 후 재확인

**검증 기준:** 개발 단계에서 작성한 **기능 검증 체크리스트**(`docs/004_planning/{프로젝트명}_기능검증_체크리스트.md`)의 모든 항목을 브라우저에서 재검증한다.

**PASS/FAIL 기준:**
- 동작하지 않는 기능 = **상** 이슈
- 동작하지만 결과가 잘못된 기능 = **중** 이슈
- 상 0건 + 중 0건이어야 PASS

## 3-D. 디자인 검증

**담당:** 디자인팀 (김지수)

디자인 산출물과 실제 구현 화면을 브라우저에서 비교한다.

**절차:**

1. 디자인 HTML을 로컬 서버로 띄운다 (`python3 -m http.server 8899`)
2. 실제 구현을 dev 서버로 띄운다 (`pnpm web:dev`)
3. 모든 화면을 모바일 뷰포트(393x852)에서 나란히 비교한다
4. 각 요소의 CSS를 브라우저 inspect로 측정하여 대조한다

**검증 항목:**

| 항목 | 확인 내용 |
|------|----------|
| 배경색 | 페이지/섹션별 배경색 일치 |
| 타이포그래피 | font-size, font-weight, line-height, color |
| 간격 | margin, padding (4px 그리드 준수) |
| 카드/컴포넌트 | border-radius, shadow, 내부 padding |
| 색상 | primary, secondary, destructive, text 색상 |
| 아이콘 | 스타일, 크기, 색상 |
| 레이아웃 | 전체 구조, 정렬, 높이 |
| 버튼 | 크기, 색상, 모서리, disabled 상태 |
| 상태 | 정상/로딩/빈/에러 4가지 상태 |

**불일치 발견 시:** 심각도(상/중/하) 분류 후 프론트팀에 수정 지시. 수정 완료 후 디자인팀이 재확인.
