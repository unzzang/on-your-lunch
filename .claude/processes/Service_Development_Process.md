# 서비스 개발 프로세스 (마스터)

> 적용 범위: 신규 서비스의 아이디어 → 출시까지 전체 흐름
> 책임자: PO (의사결정), 위더 (조율, 필요 시 호출)
> 선행 조건: 없음 (최상위 프로세스)

---

## 1. 역할

서비스 개발의 전체 Stage를 정의하고, 각 Stage의 진입 조건과 완료 기준을 관리한다.

## 2. 프로세스

```
Stage 0. 프로젝트 셋업       → Project_Setup_Process.md
Stage 1. 사전조사             → Research_Process.md
Stage 2. 기획                 → Planning_Process.md
Stage 3. 디자인               → Design_Process.md
Stage 4. 개발                 → Development_Process.md
Stage 5. QA                   → QA_Process.md
Stage 6. 출시                 → Launch_Process.md
```

### 핵심 원칙

| 원칙 | 설명 |
|------|------|
| Stage Gate | 각 Stage의 완료 기준을 충족해야 다음 Stage 진입 |
| 3라운드 정제 | 초안 → 검증(송현아) → 확정. 주요 산출물에 적용 |
| PO 직접 지시 | PO가 에이전트에게 직접 지시. 위더는 호출 시에만 참여 |
| 의사결정 기록 | 모든 주요 결정은 `.docs/001_decision-making/`에 DR로 기록 |
| Stage 검증 | 각 Stage 완료 시 1차 전문 검증 → 2차 정합성 검증 → PO 승인 |

### 팀 구조 (8개 팀, 22명)

팀 구성과 에이전트 상세는 `agents/` 참조.

### 콘텐츠/마케팅 (Stage와 독립)

- 콘텐츠 제작 → Content_Process.md
- 마케팅 전략/운영 → Marketing_Process.md

## 3. 검증 절차

각 Stage 완료 시 아래 검증 흐름을 거친다. `rules/common/stage-verification.md` 참조.

### 검증 흐름

```
담당 에이전트 작업 완료
    ↓
[1차] 전문 검증 자동 투입 (해당 Stage에 전문 검증자가 있을 때)
    ↓ 통과
[2차] 정합성 검증 자동 투입 (송현아 또는 진도준)
    ↓ 통과
[보고] PO에게 검증 결과 보고 (텔레그램 알림)
    ↓
[최종] PO 승인 (또는 위더 대리 승인)
    ↓
Stage 완료
```

### Stage별 검증 담당

| Stage | 1차 전문 검증 | 2차 정합성 검증 | 최종 승인 |
|-------|-------------|---------------|----------|
| Stage 0. 프로젝트 셋업 | — | 송현아 | PO (부재 시 위더) |
| Stage 1. 사전조사 | — | 송현아 | PO (부재 시 위더) |
| Stage 2. 기획 | 이명환 (기술 스펙 타당성) | 송현아 (산출물 정합성) | PO (부재 시 위더) |
| Stage 3. 디자인 | 김지수 (디자인 원칙 준수) | 송현아 (체크리스트 검증) | PO (부재 시 위더) |
| Stage 4. 개발 | 이명환 (코드 리뷰) | 진도준 (빌드/테스트) | PO (부재 시 위더) |
| Stage 5. QA | — | 진도준 | PO (부재 시 위더) |
| Stage 6. 출시 | — | 송현아 + 진도준 (공동) | PO (부재 시 위더) |

### 검증 실패 시

- 실패 사유를 담당 에이전트에게 전달
- 담당 에이전트가 수정 완료 후 재검증 요청
- 재검증은 실패한 검증 단계부터 재시작

### 검증 결과 보고서

검증 완료 시 `.docs/000_inbox/{날짜}({요일})_Stage{N}_검증결과.md`에 저장한다.
