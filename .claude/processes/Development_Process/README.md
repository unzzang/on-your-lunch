# 개발 프로세스 (Stage 4)

> 적용 범위: 아키텍처 결정부터 QA 투입 준비까지
> 책임자: PO (의사결정), 개발팀 (구현)
> 선행 조건: Stage 2 Gate 3 통과 + Stage 3 완료

## "동작한다"의 정의

빌드가 통과하는 것이 아니라, **브라우저에서 사용자가 기능을 조작하여 결과를 확인할 수 있는 상태.**

## Phase 흐름

```
Phase 1. 개발 준비 (1회)          → Phase1_개발준비.md
    ↓
Phase 2. 기능 단위 개발 (반복)     → Phase2_기능개발.md
    ↓
Phase 3. QA 투입 준비 (1회)       → Phase3_QA투입준비.md
```

## Stage 4 완료 기준

- [ ] Phase 1 산출물 완료 (아키텍처 DR, 환경 셋업, API 계약)
- [ ] Phase 2 전체 기능 개발 완료 (기능별 브라우저 확인 포함)
- [ ] Phase 3 Definition of Done 전체 충족
- [ ] 기능 검증 체크리스트 전체 체크
- [ ] QA 투입 가능 상태

## 위임 시 필수 지시사항

**백엔드팀:**
- "curl로 정상/에러 응답을 확인하고 결과 첨부"
- "스키마 필드가 빠짐없이 사용되는지 확인"

**프론트팀:**
- "브라우저에서 해당 기능을 직접 조작하여 동작 확인"
- "동작 전/후 스크린샷 첨부"
- "빌드 통과만으로 완료 보고 불가"

## 브라우저 검증 역할 분담

| 작업 | 담당 | 사유 |
|------|------|------|
| 브라우저 UI 인터랙션 검증 | **위더** | 서브 에이전트는 Bash 권한 제한으로 브라우저(browse) 실행 불가. 위더가 검증 수행. |
| 검증에서 발견된 코드 수정 | **해당 개발팀** | 코드 작성은 위더가 하지 않음. 이슈 목록을 정리하여 개발팀에 위임. |

**흐름:**
```
위더: 브라우저에서 기능 검증 → 이슈 발견
    ↓
위더: 이슈 목록 정리 (화면, 현상, 기대 동작)
    ↓
개발팀: 코드 수정
    ↓
위더: 브라우저에서 수정 확인
```
